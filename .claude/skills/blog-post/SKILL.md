블로그 글 초안을 질의응답으로 만드는 스킬.

> **권장 모델:** Sonnet 4.6. 글쓰기 품질은 Opus와 큰 차이 없고 비용·속도가 유리함. 현재 모델이 Opus면 `/model sonnet` 으로 전환 후 진행 권장.

## 사용법

```
/blog-post <주제>
```

예: `/blog-post Docker 입문`

## 실행 흐름

### 1단계: 질의응답 (9개 질문)

주제를 받으면 아래 9개 질문을 **한 번에** 출력한다. 사용자가 답하면 2단계로 넘어간다.
답이 빈 항목은 "[작성 필요]"로 표시하고 진행한다.

```
주제: {주제}

글을 풍부하게 쓰려면 몇 가지 물어봐야 한다.

[맥락]
Q1. 이 주제를 다루게 된 계기가 뭔가? (어떤 문제를 만났나, 어떤 필요가 있었나)
Q2. 이 주제가 지금 왜 중요한가? (배경, 트렌드, 실무에서 왜 필요해졌나)

[기술 핵심]
Q3. 기술적으로 핵심 포인트 1~3가지를 꼽는다면?
Q4. 이 기술과 비교할 만한 대안/경쟁 기술이 있나? 있다면 어떻게 다른가?
Q5. 대표적인 코드/명령어/설정 예시를 하나 보여줄 수 있나? (없으면 스킵)

[경험]
Q6. 이 주제를 다루면서 막혔거나 삽질했던 경험이 있나?
Q7. 그 문제를 어떻게 해결했나?
Q8. 입문자가 흔히 빠지는 함정이나 실전 팁 1~3가지는?

[메시지]
Q9. 이 글을 읽는 입문자에게 가장 전달하고 싶은 한 문장은?
```

### 1-a단계: 선택 섹션 결정 (내부 판단)

답변을 본 후 아래 기준으로 선택 섹션 포함 여부를 판단한다:

- Q2 답변이 풍부하면 → **"왜 지금 이 주제인가"** 섹션 추가 고려
- Q4 답변에 비교 대상이 2개 이상이면 → **"대안 비교"** 섹션 추가 고려 (표 필수)
- Q8 답변이 3개 가까이 구체적이면 → **"실전 팁 / 함정 피하기"** 섹션 추가 고려 (체크리스트)

셋 중 **최대 1개만** 선택한다. 모두 약하면 선택 섹션 없이 기본 2섹션으로 간다.

### 2단계: 마크다운 초안 생성

답변을 바탕으로 CLAUDE.md의 규칙에 따라 초안을 생성한다.

**파일명**: `YYYY-MM-DD-주제-슬러그.md` (오늘 날짜 사용)  
**저장 위치**: 프로젝트 루트 `posts/` 폴더

**구조** (선택 섹션 없음):
```markdown
# {대제목}

## {기술 설명 섹션 제목 — 질문형 권장}

### {소제목1}
### {소제목2}
### {소제목3}
### {소제목4 (선택)}

## {개발 경험 섹션 제목 — 1인칭 서술형}

### {경험 소제목1}
### {경험 소제목2}
### {경험 소제목3}
### {경험 소제목4 (선택)}

## 마치며
```

**구조** (선택 섹션 1개 포함):
```markdown
# {대제목}

## {기술 설명 섹션 제목}
### ... (H3 3~4개)

## {선택 섹션 제목 — "왜 지금 ~인가" / "~와 ~의 차이" / "~할 때 주의할 점"}
### ... (H3 2~3개, 대안 비교면 표 필수, 실전 팁이면 체크리스트)

## {개발 경험 섹션 제목}
### ... (H3 3~4개)

## 마치며
```

**포함 요소 체크리스트:**
- [ ] 코드 블록 1개 이상 (언어 명시 + 주석)
- [ ] 비교 표 1개 이상 (섹션 1 또는 선택 섹션)
- [ ] 체크리스트/요약 상자 1개 이상 (섹션 2 또는 선택 섹션)
- [ ] 전문 용어 첫 등장 시 괄호 설명

### 3단계: 이미지 프롬프트 생성

초안 하단에 아래 형식으로 이미지 프롬프트를 추가한다.
이 프롬프트는 `/blog-images` 스킬이 Replicate Flux schnell로 자동 생성한다.

```markdown
---
## 이미지 프롬프트

**대표 썸네일**
> {주제를 시각적으로 표현하는 영문 프롬프트}

**섹션 1 삽화**
> {기술 개념을 설명하는 다이어그램 또는 일러스트 프롬프트}

**섹션 2 삽화**
> {개발 경험/삽질 장면을 표현하는 프롬프트, developer at computer, frustrated then happy}
```

**이미지 프롬프트 작성 규칙:**
- **텍스트/문자를 이미지 안에 넣지 않는다** — 확산 모델이 한글/영문 텍스트를 제대로 그리지 못함. 아래 단어를 프롬프트에 쓰지 말 것:
  - `labeled`, `with text`, `sign`, `word`, `letter`, `logo`, `title`, `caption`, `writing`, `typography`
  - `"CLAUDE.md" glowing` 같이 파일명/이름을 직접 노출시키는 표현도 금지
- 영문으로 작성
- 스타일 suffix(`flat 2D illustration, soft pastel colors, ...`)는 스크립트가 자동 부착하므로 프롬프트에는 내용만
- 선택 섹션이 있으면 `**선택 섹션 삽화**` 항목을 추가
- 해당 주제/섹션을 **상징하는 장면**을 묘사 (파일명 대신 "document icon in folder tree", 코드 대신 "abstract code blocks" 식으로)

## 규칙

- CLAUDE.md의 문체, 구조, 분량 규칙을 반드시 따른다
- 이모지 사용 안 함
- 전문 용어 첫 등장 시 괄호로 설명 추가
- 코드 블록 사용 시 언어 명시 및 주석 포함
- 질문에 답이 없는 항목은 "[작성 필요]"로 표시하고 넘어간다
