# 블로그 글쓰기 규칙

## 대상 독자
- AI에 관심 있는 모든 사람, 일반인
- 전문 용어는 반드시 한 줄 설명 추가
- 비유와 예시를 적극 활용

## 제목 구조
- 대제목(H1): 주제 전체를 아우르는 한 문장
- 소제목(H2): 각 섹션 제목 — 내용을 직접 표현하는 문장으로 작성
- 소소제목(H3): 섹션 내 세부 항목
- "기술 설명", "개발 경험 공유", "오늘 배운 것" 문구를 제목에 그대로 쓰지 않는다
- 예시:
  ```
  # Docker란 무엇인가

  ## 컨테이너는 왜 등장했나       ← (기술 설명 섹션)
  ### 컨테이너가 뭔가
  ### 왜 써야 하나

  ## 나는 Docker를 이렇게 망했다  ← (개발 경험 공유 섹션)
  ### 처음 Docker를 접했을 때
  ### 삽질했던 순간

  ## 마치며
  ```

## 글 구조

기본 구조: **기술 설명 → [선택 섹션] → 개발 경험 공유 → 마치며**

선택 섹션은 주제에 맞을 때만 1개 추가한다. 기본 2섹션만으로 충분하면 넣지 않는다.

### 섹션 1: 기술 설명
- 개념 → 왜 필요한가 → 어떻게 동작하나 순서
- 소소제목(H3) 3~4개
- 코드 예시 1개 이상 (코드가 의미 있는 주제면 필수), 주석 포함
- 소제목은 질문형으로 ("왜 이걸 써야 하나")

### 선택 섹션 (주제에 맞으면 하나 선택)
- **왜 지금 이 주제인가** — 배경, 트렌드, 실무에서 왜 중요해졌는가
- **대안 비교** — 경쟁 기술/방식과 어떻게 다른가
- **실전 팁 / 함정 피하기** — 흔히 빠지는 함정

### 섹션 2: 개발 경험 공유
- 1인칭으로 작성 ("나는 ~했다", "처음엔 ~인 줄 알았는데")
- 소소제목(H3) 3~4개
- 막혔던 부분, 삽질한 것, 깨달은 것 포함
- 독자가 같은 실수를 피할 수 있게

### 마치며
- 3줄 이내 요약
- 요약 형식으로 작성, "오늘 배운 것:" 문구는 쓰지 않는다
- 다음 단계 한 가지 제안

## 문체
- 평서체 (~했다, ~이다, ~한다, ~였다)
- 짧은 문장 (한 문장 40자 이내 권장)
- 어려운 단어 대신 쉬운 단어
- 이모지 사용 안 함

### AI 글 같은 표현 금지
- "첫째, 둘째, 셋째" 나열 구조 금지
- "크게 N가지다", "크게 나누면" 식 도입 금지
- "이를 통해", "따라서", "결론적으로", "핵심은", "중요한 것은" 금지
- **볼드** 남용 금지 — 한 문단에 1개 이하
- 목록(\-) 나열보다 자연스러운 문장 흐름으로 연결

### 검색 결과 인용 규칙
- 검색으로 가져온 정보를 확인 없이 단정적 사실로 쓰지 않는다
- 확인되지 않은 수치·주장은 반드시 출처 표기 또는 제외
- 커뮤니티 반응·사용자 의견은 "~라는 반응이 있다", "~로 알려졌다" 식으로 표현
- AI가 알고 있는 것도 검색 결과와 다를 수 있다. 검색 결과와 충돌하면 검색 결과 우선

## 파일명 규칙
- 형식: `YYYY-MM-DD-주제-슬러그.md`
- 예: `2026-04-19-docker-입문.md`

## 분량
- 총 2,500자 ±10% (2,250~2,750자, 마크다운 기준, 코드블록·이미지 프롬프트 제외)
- 섹션 1(기술): 1,000자
- 선택 섹션(있을 때): 500자
- 섹션 2(경험): 800자
- 마치며: 200자
- 선택 섹션이 없으면 섹션 1/2가 각각 +200자씩 늘어나도 됨
