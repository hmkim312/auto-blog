블로그 파이프라인. 주제 탐색 → 초안 → 사용자와 H3 단위 피드백 → 가벼운 검수 → 카드 생성 순서로 진행한다.

> **권장 모델:** Sonnet 4.6. 글쓰기 품질은 Opus와 큰 차이 없고 비용·속도가 유리함. 현재 모델이 Opus면 `/model sonnet` 으로 전환 후 진행 권장. 특히 2단계(H1/H2/H3 피드백)는 반복 호출이 많아 Opus면 비용이 빠르게 쌓인다.

## 사용법

```
/blog                              # 주제 탐색부터 시작
/blog uv 패키지 매니저              # 주제 직접 지정
/blog --profile hmkim
/blog uv --profile hmkim
```

## 실행 흐름

### 1단계: 초안 생성 (blog-research 위임)

`blog-research` 스킬의 전체 흐름을 그대로 수행한다:

1. CLAUDE.md 규칙 및 `blog-research` SKILL.md 규칙을 적용
2. **주제 탐색** (`scripts/discover-topics.py` → 후보 10개 → 사용자 선택) → 주제 확정
3. **핵심 키워드·슬러그 결정** (0.5단계) — 한국 검색 트렌드 기준 후보 3~5개 추천 → 사용자 선택 → 한글 슬러그 생성 → 파일명 확정. 이 키워드는 H1·도입부·썸네일 alt 에 동일 형태로 들어간다.
4. 세션 ID 생성 → 프로파일 로드 → AI 쿼리 제안 → 사용자 승인 → Tavily 검색 (≤6 쿼리)
5. 심화 키워드 1~2회 (선택, 총 Tavily 호출 ≤8)
6. **검색 맞춤 Q&A** 를 한 번에 출력하고 사용자 답변 대기
7. `posts/YYYY-MM-DD-<핵심 키워드 슬러그>.md` 에 초안 + 인라인 출처 작성. **`## 이미지 메타` 블록은 이 단계에서 생성하지 않는다** — H3 피드백 완료 후 4단계에서 만든다.

### 2단계: 플로우 설명 + 피드백

**모델 체크 (필수)**: 현재 모델이 Opus면 사용자에게 "피드백 단계는 반복 호출이 많아 Sonnet 권장. `/model sonnet` 으로 바꾸시겠어요?" 안내. 사용자 결정 받고 진행.

초안 파일 경로를 밝히고, 먼저 전체 글의 구조와 의도를 설명한다:

**2-A. 전체 플로우 설명**

다음 형식으로 글 전체 흐름을 한 번에 보여준다:

```
H1: <제목> — <이 제목을 고른 이유>
H2 1: <제목> — <이 섹션이 다루는 것>
  H3: ...
  H3: ...
H2 2: <제목> — <이 섹션이 다루는 것>
  ...
마치며: <제목> — <한 줄 결론 방향>

전체 흐름: <독자가 읽으면 어떤 순서로 무엇을 알게 되는지 2~3줄>
```

이 설명을 보고 사용자가 구조 수정 요청을 하면 반영한 뒤 다음 단계로.

**2-B. 전체 초안 작성 → 한 번에 피드백**

구조 확정 후 전체 초안을 한 번에 작성해 텍스트로 보여준다. H3 단위로 쪼개서 OK를 받지 않는다.

1. 전체 초안(H1·도입부·모든 H2·H3·마치며)을 작성해 텍스트로 출력
2. 사용자 피드백 대기 — 수정 요청이 오면 해당 부분만 반영
3. 최종 확인 후 파일에 한 번에 저장 (Edit/Write)
4. 완료 후 3단계로

**작성 시 자가 체크 (필수, 출력 전에 통과)**

`/blog-research` 11단계의 자가 체크 항목 전체를 똑같이 적용하되, 본 단계에서 다시 강조할 두 항목:

- **H2 도입 단락 금지** — 본문 H2 직후는 곧장 H3 (또는 이미지 마커 → H3) 로 진입. H2 와 H3 사이에 별도 단락을 끼우지 않는다. 도입 단락은 H1 다음 두 단락에만. 마무리 H2 는 H3 없이 본문만 두는 게 정상. 사용자 피드백 자리 줄이는 사전 차단 룰. CLAUDE.md `## 글 구조` 룰.
- **자연 한국어로 처음부터 작문** — 영문 검색 결과를 그대로 옮기지 않는다. 사실·수치·인용만 가져오고 본문은 한국어 어순·동사 선택으로 처음부터 짠다. 영어식 직번역 구문이 글에 나오면 즉시 다시 쓴다. 메모리 [feedback_blog_natural_korean] · [feedback_blog_korean_drafting] 룰.

위 두 룰은 사후 검수(3-A) 가 아니라 출력 직전에 잡는다. 검수 단계 부담을 줄이고, 사용자 피드백이 동일 룰을 두 번 지적하지 않도록.

**진행 방식:**
- 파일 저장은 피드백 반영 완료 후 한 번만
- 사용자가 직접 수정한 내용은 그대로 보존
- 제목·본문 둘 다 피드백 대상

### 3단계: 검수 + 자동 보강

피드백 완료 후 검수 항목 출력 + 두 가지 자동 보강(ILJ·출처)을 같은 단계에서 처리한다. 4단계(카드 메타)로 넘어가기 전에 반드시 통과.

#### 3-A. 검수 항목

- **글자수** — 총 글자수 (3,150~3,850 기준, 부족해도 크게 문제 없음)
- **오탈자** — 발견한 목록. 없으면 "없음"
- **전체 플로우** — H1→H2→마치며 흐름 어색한 곳. 없으면 "자연스러움"
- **유입 점수** — blog-review SKILL.md 19번 5축 합산 (100점 만점), 낮은 축 보강 제안 1줄
- **H2 도입 단락 점검 (필수)** — 본문 H2 직후가 `이미지 마커 → H3` 패턴인지 확인. 마무리 H2 는 H3 없이 본문만 두는 게 정상이라 제외. 이미지 마커와 H3 사이에 단락이 있으면 위반 → 통째 삭제하거나 H3 안으로 흡수. CLAUDE.md `## 글 구조` 룰.
- **번역 어투 점검 (필수)** — 본문 전체를 LLM 자가 점검으로 다시 읽어 영어식 직번역 표현을 잡아낸다. 한국어 어순·종결·동사 선택이 자연스러운지, 영문 글을 그대로 옮겨 놓은 듯한 흐름이 있는지 본다. 의심 자리는 자연 한국어 시안과 함께 사용자에게 보여주고 일괄 정정. 메모리 [feedback_blog_natural_korean] · [feedback_blog_korean_drafting] 룰.

#### 3-B. 인터럴 링크 자동 매칭 (필수)

`internal-links.md` 를 Read 로 읽고, 본문에 자연스럽게 등장하는 ILJ 키워드를 자동 매칭한다.

1. `internal-links.md` 의 "포스트 제목 → 키워드" 표 파싱
2. 본문(코드블록·alt·태그 섹션 제외)에서 각 키워드 첫 자연 등장 위치 탐지
3. 등장 키워드 목록 + 매칭 대상 글을 사용자에게 한 번에 보여주고 OK 받기:
   ```
   ILJ 키워드 N개 발견:
   - "AI 에이전트" (L42) → "AI 에이전트 입문" 글
   - "RAG" (L78) → "RAG 청킹" 글  ← 매칭 정확한지 확인 필요
   매칭 OK?
   ```
4. OK 받으면 일괄 적용 (Edit):
   - 본문 첫 등장 키워드 뒤에 `<!--ILJ-->` 마커 삽입
   - `## 태그` 바로 아래에 `## 인터럴 링크` 섹션 생성/갱신 (나가는·들어오는 링크)
   - 들어오는 링크 키워드 2개는 LLM 이 글 핵심 키워드 기반으로 제안 → 사용자 OK
   - `internal-links.md` 에 이 글 항목 추가/갱신

본문에 매칭 키워드가 0개여도 `## 인터럴 링크` 섹션은 만든다 (들어오는 링크는 항상 필요).

#### 3-C. 출처 자동 매칭 (필수)

`/blog-research` 단계에서 받은 Tavily 검색 결과를 컨텍스트로 유지한다. 본문에서 인라인 출처가 필요한 케이스를 탐지하고 URL 을 매칭한다.

탐지 케이스 (CLAUDE.md `## 검색 결과 인용 규칙` + `/blog-research` 11단계):
- (a) 공식 발표·공식 문서 인용 — "앤트로픽이 공개", "X사가 발표" 등
- (b) 원문 직접 인용 — 따옴표 인용
- (c) 최신성·수치 주장 — "2024년 11월 출시", "수백 개 이상", "10배 빠르다" 등

처리:
1. 본문 스캔으로 후보 문장 추출
2. Tavily 검색 결과 URL 과 의미 매칭
3. 누락 목록 + 매칭 URL 을 사용자에게 한 번에 보여주고 OK 받기:
   ```
   인라인 출처 누락 N건:
   - L24 "앤트로픽이 공개한 오픈 프로토콜" → anthropic.com/news/model-context-protocol
   - L42 "수백 개 이상의 MCP 서버" → github.com/punkpeye/awesome-mcp-servers
   - L78 "10배 빠르다" → 매칭 없음  ← 수동 처리 필요
   추가 OK?
   ```
4. OK 받으면 인라인 `(출처: URL)` 일괄 삽입 (Edit)
5. 매칭 실패 항목은 "수동 확인 또는 완곡 표현(`~로 알려졌다`)으로 변경 필요" 로 알림
6. 평범한 설명 문장에는 출처를 붙이지 않는다 (오버 첨부 방지)

### 4단계: 카드 메타 생성

검수 완료 후 `/blog-research` SKILL.md 13단계 규칙을 따라 `## 이미지 메타` 블록을 새로 생성:

- 팔레트 3색 (글 톤·직전 글과 겹치지 않게)
- thumb 1개 + section-N (마무리 제외 본문 H2 수만큼, 최대 4개)
- 각 카드 타이틀은 **피드백 완료된 본문 H2 내용과 정렬**
- **alt 자동 작성** — 카드별 30~80자 한국어 묘사. thumb alt 에만 0.5단계 핵심 키워드 자연 포함, section-N alt 는 해당 H2 메시지 중심. 카드 간 alt 중복 금지.
- 생성 후 사용자에게 한 번에 보여주고 "수정할 거 있냐" 확인. 한 장씩 OK 받지 않음.
- 확정된 alt 는 본문 마크다운 `![alt](path)` 의 alt 자리에 동기화 (Edit 으로 갱신).

### 5단계: 카드 이미지 생성

```
uv run scripts/render-cards.py posts/<파일>
```

생성 완료 후 파일 경로 안내.

## 규칙

- 자동 검수/교정(blog-review --fix) 은 실행하지 않는다.
- H3 피드백은 텍스트 시도 → OK → Edit 적용 순서. OK 받기 전에 파일에 쓰지 않는다.
- 카드 이미지는 카드 메타 확인 후 생성.
- Tavily 키(`TAVILY_API_KEY`)가 `.env` 에 없으면 1단계에서 중단.

## 에러 대응

- 1단계에서 사용자가 중단하면 2단계로 진행하지 않음
- 카드 렌더 실패 시 해당 카드만 재시도: `/blog-cards <파일> --only <key>`
