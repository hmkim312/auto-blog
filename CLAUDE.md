# 블로그 글쓰기 규칙

## 대상 독자
- AI에 관심 있는 모든 사람, 일반인
- 전문 용어는 반드시 한 줄 설명 추가
- 비유와 예시를 적극 활용

## 제목 구조

대제목·소제목은 **그 섹션에서 실제로 무슨 이야기를 하는지 한 줄로 보이는 문장**으로 짓는다. 독자가 제목만 훑어도 글의 흐름이 이해돼야 한다.

- 대제목(H1): 주제를 한 문장으로. 검색 키워드가 우선이라 정의형(`~란?`, `~가 뭔가`)도 허용.
- 소제목(H2): 그 섹션의 결론·이야기 축을 직접 드러내는 문장.
- 소소제목(H3): 섹션 내 세부 단락. 구체적인 명사·동사 포함.

### 금지 패턴 (AI 템플릿 냄새)
- `~가 뭔가`, `~는 무엇인가`, `~란 무엇인가` 같은 정의형 공식 (H2·H3 한정. H1 은 검색 키워드 우선이라 허용)
- `왜 ~인가`, `왜 써야 하나` 같은 내용 없는 질문 공식 (단, 그 섹션이 실제로 "왜"에 답한다면 허용)
- `어떻게 동작하나`, `어떻게 쓰나` 같은 추상적 공식
- `기술 설명`, `개발 경험 공유`, `오늘 배운 것` 문구를 제목에 그대로 쓰지 않는다
- 같은 글 안에서 질문형 제목을 3개 이상 반복하지 않는다 (템플릿감 유발)

### 좋은 제목 기준
- 명사 하나로 끝내지 말고 **동사·비유·결과**를 한 조각 얹는다
- 독자가 "어 이건 내 얘기네" 하고 걸려들 만한 구체 장면
- 제목만 모아 읽어도 글의 줄거리가 이어진다

### 글 전체는 정보 전달 톤으로 일관한다

블로그 글의 H1·H2·H3 는 정보 전달 톤으로 묶어 둔다. `포켓몬 하는 클로드를 보다가 무서워졌다`, `~하다 깨달았다`, `~던 날` 같은 회고·1인칭 후크형 제목은 본문 헤딩에 쓰지 않는다.

도입부 두 번째 단락의 1인칭 후크는 그대로 살린다 (CLAUDE.md `## 도입부` 룰). 그 자리 외에는 안내·정리 어조로 일관한다.

자연스러운 정보 전달 제목은 검색·읽기 동선에 가까운 한국어 어순으로 짠다.
- `~할 때 조심할 것`, `~ 처음 쓸 때 막히는 지점`
- `~ 차이`, `~ 한계`
- `~하는 방법`, `~할 때 알아둘 것`

### 정보 전달 톤일 때 같이 피할 패턴 (AI 정보글 냄새)

정보 전달 톤이라는 말이 모든 제목을 콜론 부제·숫자 정형구로 깎으라는 뜻은 아니다. AI 정보글 특유의 정형구로 흐르면 그것대로 톤이 깨진다. 한 글 안에서 다음 패턴이 반복되면 다시 푼다.

- **콜론 부제 남용** — `권한 부여 방식: MCP 기반 폴더 화이트리스트` 같은 `제목: 명사구` 형식이 한 글의 H3 절반을 넘으면 AI 템플릿감이 짙어진다. 자연 한국어 어순으로 풀어 쓴다.
- **추상 명사 마무리** — `~ 정리`, `~ 대응`, `~ 가이드`, `~ 활용`, `~ 한눈에 보기`, `~ 톺아보기` 같은 마무리는 한 글에 1개 이하.
- **`N가지 X` 정형구** — `세 가지 경로`, `네 가지 방법`, `다섯 가지 팁` 같은 숫자 강박은 한 글에 1개 이하. 본문이 진짜 N개로 쪼개질 때만 쓴다.
- **명사 나열형** — `사용 영역`, `권한 구조`, `보호 장치` 같은 영어 직역식 명사 묶음은 한국어로 풀어 쓴다.
- **반복되는 종결** — H2·H3 를 4~5개 나란히 두고 읽었을 때 종결·어조가 다양해야 한다. 전부 `~ 방식`, `~ 구조`, `~ 정리` 식으로 맞물리면 AI 냄새.

기준은 단순하다. 제목을 소리 내 읽었을 때 한국어 글쓰기 책에서 본 적 있는 말투인지, 검색 결과 자동완성에 나올 법한 어색한 명사 묶음인지. 후자면 다시 푼다.

### 예시 (개선 방향)

Before (템플릿):
```
# Docker란 무엇인가
## 컨테이너는 왜 등장했나
### 컨테이너가 뭔가
### 왜 써야 하나
```

After (정보 전달 톤, 자연 한국어 어순):
```
# Docker 처음 쓸 때 알아두면 좋은 것

## 컨테이너가 어떤 식으로 돌아가는지부터 본다
### 프로세스를 격리해서 들고 다닐 수 있다
### VM 과는 무게부터 다르다
### docker run 한 줄로 환경이 따라온다

## 입문할 때 자주 막히는 지점
### 이미지가 10GB 로 부풀어 오를 때
### latest 태그만 믿었다가 배포가 깨질 때
### 캐시 레이어 이해하고 빌드 속도 잡기

## 마치며
```

`VM 과는 무게부터 다르다` 처럼 평서형이 자연스럽고 구체적이면 좋다. 같은 글에서 `~가 뭔가` 식 공허한 질문 공식이 반복되거나, 모든 H3 가 콜론 부제 형식으로 깎이는 패턴은 피한다.

## 도입부

H1 → 썸네일 카드 → 두 단락 → 첫 H2 순서로 둔다. 검색엔진은 이미지를 건너뛰고 다음 텍스트를 description 으로 잡으므로, 썸네일 다음 두 단락이 description 자리이자 독자 진입 동선이다.

> **주의:** `## 도입부` 라는 H2 제목은 본문에 쓰지 않는다. 이 절은 룰 문서 안에서만 쓰는 구분 이름이고, 실제 글에는 H1 바로 밑에 썸네일 → 두 단락만 두고 곧장 첫 진짜 H2(기술 설명)로 넘어간다.

### 첫 단락 — SEO 정의 한 줄
- 길이 100~160자 (1~2문장)
- "누가 / 언제 / 무엇을 / 왜 중요한가" 중 2~3개를 담은 정의문
- 제목의 핵심 키워드 자연스럽게 포함
- 평서체. **1인칭 금지**. 볼드·이모지 없음
- 검색 결과에 그대로 노출되는 줄이라 담백하게

### 두 번째 단락 — 1인칭 후크
- 기존 회고 톤 그대로 (구체 장면 1인칭)
- 2~4문장
- 본문으로 끌어들이는 역할

### 형식
- 썸네일 카드 마커를 H1 **바로 아래** 에 둔다 (제목 → 이미지 → 본문 흐름, 가독성 우선). alt 는 30~80자 한국어 묘사(`## SEO` 절).
- 그 다음에 두 단락(SEO 정의 → 1인칭 후크) 을 둔다. 단락 사이 빈 줄 1개로 분리
- 두 단락 합쳐 **250자 이상**이 되도록 둔다 (description에 H2 가 끌려 들어오는 사고 방지)
- 두 단락 다음에 곧장 첫 H2 로 넘어간다 (워드프레스에서 광고는 자동 삽입이라 본문 마커 둘 필요 없음).

### 예시

```
# 발표 슬라이드에 일주일 갈아 넣었던 사람의 클로드 디자인 첫 인상

![클로드 디자인 첫 인상 — 발표 자료 만들다 마주친 AI 디자인 도구](images/2026-04-26-클로드-디자인/thumb.webp)

클로드 디자인(Claude Design) 은 엔트로픽 랩스(Anthropic Labs) 가 2026년 4월 17일 공개한 AI 디자인 도구다. 프롬프트 한 줄로 와이어프레임과 시안을 생성해 주며, 기반 모델은 오푸스 4.7(Claude Opus 4.7) 이다.

발표 자료를 만들 때마다 디자인이 발목을 잡았다. 글자 크기, 색 조합, 정렬 하나에 시간이 녹아내렸다. 그래서 신제품 발표 소식을 봤을 때 일단 정리부터 해 보고 싶었다.

## 프롬프트 한 줄로 와이어프레임을 만든다는 도구가 나왔다
```

### 왜 이 위치인가
검색엔진(구글·네이버 등)이 `<meta name="description">` 자동 생성 시 이미지를 건너뛰고 다음 텍스트부터 약 155자를 잘라 쓴다. 썸네일 다음 첫 단락이 정확히 그 자리다. 정의 한 줄을 두면 description 이 그 한 문장으로 깔끔하게 잡히고, 가독성도 제목 → 이미지 → 본문 순서로 자연스럽다.

## 글 구조

기본 구조: **기술 설명 → [선택 섹션] → 개발 경험 공유 → 마치며**

선택 섹션은 주제에 맞을 때만 1개 추가한다. 기본 2섹션만으로 충분하면 넣지 않는다.

### 섹션 1: 기술 설명
- 개념 → 왜 필요한가 → 어떻게 동작하나 순서
- 소소제목(H3) 3~4개
- 코드 예시 1개 이상 (코드가 의미 있는 주제면 필수), 주석 포함
- 소제목은 위의 "제목 구조" 규칙을 따른다. 질문형 템플릿 반복 금지.

### 선택 섹션 (주제에 맞으면 하나 선택)
- **왜 지금 이 주제인가** — 배경, 트렌드, 실무에서 왜 중요해졌는가
- **대안 비교** — 경쟁 기술/방식과 어떻게 다른가
- **실전 팁 / 함정 피하기** — 흔히 빠지는 함정

### 섹션 2: 개발 경험 공유
- 1인칭으로 작성 ("나는 ~했다", "처음엔 ~인 줄 알았는데")
- 소소제목(H3) 3~4개
- 막혔던 부분, 삽질한 것, 깨달은 것 포함
- 독자가 같은 실수를 피할 수 있게

### 마무리 (H2 제목 화이트리스트 중 1개)

마무리 섹션의 H2 제목은 글 톤에 맞게 아래 8개 중 하나를 고른다. 화이트리스트 외 제목은 쓰지 않는다.

```
마치며 / 끝으로 / 나가며 / 한 문장으로 / 한 줄로 남긴다면 / 마지막 한 줄 / 짧게 정리 / 결국 남는 건
```

- 차분한 글: 마치며 / 끝으로 / 나가며
- 한 줄 결론형: 한 문장으로 / 한 줄로 남긴다면 / 마지막 한 줄 / 짧게 정리
- 회고·감정형: 결국 남는 건

내용 규칙:
- 3줄 이내 요약
- "오늘 배운 것:" 문구는 쓰지 않는다
- "다음 글에서 다루겠다" 식 후속 약속 금지
- 다음 단계 한 가지 제안 (있으면)

## 문체
- 평서체 (~했다, ~이다, ~한다, ~였다)
- 짧은 문장 (한 문장 40자 이내 권장)
- 어려운 단어 대신 쉬운 단어
- 이모지 사용 안 함

## 톤 가드 — 키워드 강박·경험 윤색 금지

- **한 키워드 강박 금지** — 핵심 수치·단어가 H1부터 카드까지 모두 박히면 본질이 가려진다. 그 키워드는 H1 + 도입 첫 단락 + 썸네일에만 두고, H2·H3·본문 카드 타이틀은 본질·메커니즘·교훈을 직접 이야기한다.
- **본인 경험 윤색 금지** — 본인 경험·도입 사례는 실제 답변에 있는 것만 적는다. 도입 안 한 가드·도구·단계를 본문에 채우지 않는다. 사실이 약하면 짧게 마치되 빈 곳을 가짜로 채우지 않는다.
- **카드 ↔ 본문 H2 정렬** — 본문 카드의 타이틀은 그 H2 의 메시지와 정렬되어야 한다. 카드 라벨은 의미 있을 때만 쓴다 (비대칭 라벨·빈 라벨 금지).

## 영어 용어 표기

영어 고유명사·기술 용어는 **한글로 음역**해서 쓴다. 첫 등장에만 괄호 안에 영문 원문을 병기하고, 두 번째부터는 한글만 쓴다.

- 첫 등장: `클로드 디자인(Claude Design)`, `바이브 코딩(Vibe Coding)`, `엔트로픽 랩스(Anthropic Labs)`, `피그마(Figma)`
- 두 번째 이후: `클로드 디자인은…` (영문 병기 X)
- 코드·명령어·URL·파일명·태그명은 그대로 영문 유지
- 한글 음역이 어색한 약어(API, GPU, CLI, JSON 같은 머리글자)는 영문 그대로 사용 가능
- 모델명·버전명도 첫 등장 시 한글+영문 병기 권장 (`오푸스 4.7(Claude Opus 4.7)`)

## 가독성 — 문단 나누기 (필수)

블로그는 화면에서 읽히므로 긴 덩어리 글은 반드시 쪼갠다.

- **한 문단 2~4문장**을 원칙. 5문장 넘어가면 끊을 지점을 찾는다.
- **문단과 문단 사이는 빈 줄(블랭크 라인) 1줄**로 구분한다. 마크다운에서 줄바꿈만으로는 문단이 나뉘지 않으므로, 에디터에서는 **더블 엔터**로 친다.
- **주제·시점·어조가 바뀌는 지점**에서 무조건 끊는다. 예: 개념 설명 → 예시, 사실 나열 → 해석, 외부 인용 → 내 의견.
- 수치·인용·예시가 이어지는 경우, 각 항목마다 단락을 분리해도 된다. 나열이 길어질수록 쪼갠다.
- 한 섹션(H3) 안에 문단이 2~4개 나오는 게 기본. 하나의 긴 덩어리로 쓰지 않는다.

### AI 글 같은 표현 금지
- "첫째, 둘째, 셋째" 나열 구조 금지
- "크게 N가지다", "크게 나누면" 식 도입 금지
- "이를 통해", "따라서", "결론적으로", "핵심은", "중요한 것은" 금지
- **볼드** 남용 금지 — 한 문단에 1개 이하
- 목록(\-) 나열보다 자연스러운 문장 흐름으로 연결

### 검색 결과 인용 규칙
- 검색으로 가져온 정보를 확인 없이 단정적 사실로 쓰지 않는다
- 확인되지 않은 수치·주장은 반드시 출처 표기 또는 제외
- 커뮤니티 반응·사용자 의견은 "~라는 반응이 있다", "~로 알려졌다" 식으로 표현
- AI가 알고 있는 것도 검색 결과와 다를 수 있다. 검색 결과와 충돌하면 검색 결과 우선

## SEO — 핵심 키워드, 슬러그, 이미지 alt

검색으로 들어오는 글이라 키워드를 한 곳에 모은다. **글 1개 = 핵심 키워드 1개**.

### 핵심 키워드 1개를 정해 다음 자리에 그대로 둔다
- **H1 제목** — 자연스럽게 한 번 포함
- **URL 슬러그 / 파일명** — 한글 그대로 (공백은 하이픈)
- **도입부 첫 단락 (SEO 정의 한 줄)** — 자연 포함
- **썸네일 이미지 alt** — 자연 포함

핵심 키워드는 한국 독자가 실제로 검색창에 칠 만한 단어로 한글로 짓는다 (`Claude Design` ❌ → `클로드 디자인` ✅). 영문 약어(API, GPU 등) 가 핵심 키워드면 그대로 둘 수 있지만, 가능한 한 한글 음역.

두 명사를 조합한 키워드(예: `클라우드 온프레미스`)는 본문·제목에서 `와/과` 조사를 붙여 자연스럽게 쓴다(`클라우드와 온프레미스`). 슬러그에서는 조사를 뺀다(`클라우드-온프레미스`).

같은 키워드를 H2·H3·카드 타이틀까지 모두 박지는 않는다 (CLAUDE.md `## 톤 가드` 절 — 키워드 강박 금지). 위 4자리에만 두고, 본문 H2·H3 는 본질·메커니즘·교훈으로 다시 쓴다.

**본문 자연 등장 최소 5회** — 4자리(H1·슬러그·도입부·썸네일 alt) 외 본문 H2 일부·문장 내 지칭에서 자연스럽게 충족한다. 강박적 반복 아닌 맥락상 필요한 자리에만. 5회 미만이면 도입부·본문 일부를 보강한다.

### H1 숫자 옵션
H1 안에 숫자(연도, N가지, N분 등) 가 자연스럽게 들어가면 클릭율이 오르는 편이다 (`2026년 클로드 디자인 후기`, `클로드 디자인 후기 5가지`). 어색하면 강제하지 않는다 — 옵션이지 필수 아님.

### 이미지 alt — 카드별로 다른 한국어 묘사
- 본문 이미지(`![alt](path)`) 의 alt 는 그 카드의 실제 내용을 한국어 30~80자로 묘사한다.
- 썸네일(thumb) alt 1장에 핵심 키워드 자연 포함. 섹션 카드(section-N) alt 는 해당 H2 의 메시지 중심.
- **같은 alt 두 번 이상 사용 금지** — `이미지: 썸네일`, `이미지: section-1` 같은 자리채우기 금지.

## 파일명 규칙
- 형식: `YYYY-MM-DD-<핵심 키워드 한글 슬러그>.md`
- 한글 그대로 사용. 공백은 하이픈으로 치환. 마침표·괄호·물음표 같은 특수문자는 제거.
- 예: `2026-04-19-도커-입문.md`, `2026-04-26-클로드-디자인.md`

## 분량
- 총 3,500자 ±10% (3,150~3,850자, 마크다운 기준, 코드블록·`## 태그`·`## 이미지 메타` 제외)
- 섹션 1(기술): 1,400자
- 선택 섹션(있을 때): 700자
- 섹션 2(경험): 1,100자
- 마치며: 300자
- 선택 섹션이 없으면 섹션 1/2가 각각 +300자씩 늘어나도 됨
- 분량을 채우려 빈 말·중복·AI 투로 늘리지 않는다. 근거 없이 길어질 바에 짧게 마친다. 길이는 "검색 근거·경험·예시가 실제로 있을 때"만 늘어난다.
