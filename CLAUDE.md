# 블로그 글쓰기 규칙

한국어 기술 블로그의 톤과 절차. 이 파일이 단일 정관이고, 세부 절차·검수 기준·카드 명세는 각 SKILL.md 가 갖는다.

## 1. 글쓰기 핵심 원칙

1. **먼저 말한다** — 결론을 첫 문장에 둔다. 평론·격언으로 우회 X.
2. **짧게 쓴다** — 한 문장 40 자 이내, 한 단락 2~4 문장.
3. **능동형으로 쓴다** — `~ 됐다`, `~ 시켜진다`, `~ 가 되어 있다` 같은 수동형은 영어 직역이다.
4. **한 문장에는 한 가지 내용만** — 영어식 긴 수식·관계절·콤마 나열은 단문 둘로 끊는다.
5. **추상 주장은 숫자·예시·코드·상황으로 받친다** — 한 줄로 마치고 다음 줄에 구체 사례.

## 2. 번역과 작문 분리 — 한국어 사실 카드 먼저

영문 자료를 LLM 한 번에 한국어 글로 옮기면 어순·동사 결합·명사화가 영어식으로 따라온다. 그래서 **번역과 작문을 분리한다**.

영문 자료에서 사실·수치·인용만 뽑아 한국어 카드로 정리:

```markdown
### 카드 1: 정의
- 사실: 하네스 엔지니어링은 모델 환경의 도구·권한·피드백 루프를 짜는 분야다.
- 시점: 2026년 3월부터 표현이 자주 보였다.
- 출처: nxcode.io/...
```

규칙:
- 영어 단어·관용구 그대로 옮기지 않는다.
- 영어 수동태 → 한국어 능동형 (`is reported` → `보고했다`).
- 영어식 명사구 (`reliability problems`) → 동사형 (`결과가 흔들리는 문제`).
- 의견·해석 추가 X. 사실만.

작성자는 한국어 사실 카드만 본다. 영문 원문 노출 X.

## 3. 어색 어휘 리스트

`awkward-words.md` 가 단일 출처. 본문 검수 시 그 파일의 단어 또는 유사 표현이 검출되면 앞뒤 맥락에 맞게 수정한다. 대체어는 맥락에 맞게 판단.

## 4. 번역투 회피

1. **수동태 → 능동형** (`결과가 흔들리게 된다` → `결과가 흔들린다`)
2. **추상 명사 마무리 → 동사형** (`~ 한 분야` → 동사로 풀어 쓴다)
3. **콤마 나열 → 단문 분해** (명사 5~6 개 결합 한 문장은 단문 둘로)
4. `~ 라는 뜻이다`, `~ 부분`, `~ 영역` 남용 한 글에 2 회 이하

## 5. 골든 톤 표본 (조대협 블로그)

- **상황 제시 도입**: "Claude Code 로 복잡한 작업을 하다 보면 한 가지 문제를 마주치게 된다. ..."
- **결론 먼저**: "SubAgent 는 Claude Code 를 훨씬 더 강력하게 만드는 핵심 기능이다."
- **비유로 풀기**: "인간 세계에서는 이 문제를 이미 해결했다. 바로 전문화이다."

동사형 종결 70% 이상, 영문 기술 용어 그대로, 강조어 (`바로`, `특히`, `핵심은`) 자유, 비유 권장.

## 6. 도입부

H1 → 썸네일 카드 → 두 단락 → 첫 H2.

**첫 단락 — 두 형태 중 선택**
- **정의형** (검색 키워드형): 100~160 자, 1~2 문장, 키워드 자연 포함, 평서체, 1인칭 X
- **상황 제시형** (조대협 톤): 100~180 자, 2~3 문장, "~ 하다 보면 ~ 하게 된다" + 한 줄 결론

**두 번째 단락 — 1인칭 후크**: 2~4 문장. 본인 경험·동기·첫 인상. 메타 멘트 (`이 글에서는 ~ 다룬다`) X, 변명 X.

두 단락 사이 빈 줄 1 개, 다음 줄에 곧장 첫 H2.

## 7. 제목

대제목·소제목은 그 섹션이 실제로 무엇을 말하는지 한 줄로 드러내는 문장.

- 동사형 기본, 명사형도 OK. 명사형 강제 X.
- 정의형·의문형 (`~ 란?`, `왜 ~ 가 필요한가`) 자유. 한 글에 4 개 이하.
- 의문문 H2 다음은 본문에서 답한다.
- 금지: `기술 설명`, `개발 경험 공유`, `오늘 배운 것` 그대로.
- `~ 정리`, `~ 가이드`, `~ 한눈에 보기` 추상 마무리 한 글에 1 개 이하.
- `세 가지 경로`, `네 가지 방법` 숫자 강박 한 글에 1 개 이하.

## 8. 글 구조

기본: **기술 설명 → [선택 섹션] → 개발 경험 → 마치며**

선택 섹션은 주제에 맞을 때 1 개. 본인 경험이 없으면 경험 섹션 대신 "함정·실전 팁".

H2 직후에는 도입 단락 X. 곧장 H3 (또는 이미지 마커 → H3) 로 진입.

**마치며**: 자연스러운 문장. 본문 3 줄 이내 요약. "오늘 배운 것:" X, "다음 글에서 다루겠다" X.

## 9. 문체

- 평서체 (`~ 한다`, `~ 된다`, `~ 이다`) 기본
- 한 문장 40 자 이내
- 이모지 X
- em-dash (—) X — 쉼표·마침표·괄호로 대체
- 강조어 (`바로`, `특히`, `꼭`, `핵심은`) 한 단락 1~2 개
- 금지: "첫째, 둘째", "크게 N 가지다", "이를 통해/따라서/결론적으로" 남용, 볼드 남용

블로그 본문은 평서체. 채팅 대화는 존댓말 (반구어체 OK).

## 10. 영어 용어 표기

- 한글 음역이 자연스러우면 한글 (클로드, 챗GPT, 제미나이, 커서)
- 영문이 자연스러우면 영문 (SubAgent, MCP, JSON, GPU, IDE)
- 한 글 안에서 같은 단어 통일
- 코드·명령어·URL·파일명은 항상 영문

## 11. SEO

글 1 개 = 핵심 키워드 1 개. 본문 자연 등장 5 회 이상.

**4 곳에 키워드 자연 포함**: H1, URL 슬러그·파일명, 도입부 첫 단락, 썸네일 alt.

H2·H3 까지 강박적으로 박지 않는다.

두 명사 조합 (`클라우드 온프레미스`) 은 본문에서 조사로 결합 (`클라우드와 온프레미스`), 슬러그는 조사 빼고 (`클라우드-온프레미스`).

### 이미지 alt
- 본문 `![alt](path)` alt 는 카드 내용 한국어 30~80 자.
- 썸네일 alt 1 장에만 핵심 키워드 자연 포함.
- 섹션 카드 alt 는 해당 H2 메시지 중심.
- 같은 alt 두 번 이상 X.

### 내부 링크 (Internal Link Juicer)
- 초안 작성 전 `internal-links.md` 확인.
- 본문 ILJ 키워드 자연 등장 시 `<!--ILJ-->` 주석. 강제 X.
- `## 인터럴 링크` 섹션 (`## 태그` 다음 필수): 나가는 링크 + 들어오는 링크 키워드 2 개.

## 12. 분량

- **최소 2,000 자** (코드블록·꼬리 메타 제외). 상한 강제 X.
- 핵심 키워드 본문 자연 등장 **5 회 이상** (`## 11. SEO`).
- 섹션별 자수 배분 강제 X. 주제·메시지에 맞춰 자유.
- 분량 채우려고 빈 말·중복 X. 짧게 마쳐도 OK.

## 13. 가독성

- 한 문단 2~4 문장. 5 문장 넘으면 끊는다.
- 문단 사이 빈 줄 1 개.
- 한 H3 안에 문단 2~4 개.

## 14. 검색 결과 인용

- 검색 정보를 확인 없이 단정하지 않는다.
- 비공식 출처는 `~ 라고 짚었다`, `~ 라는 분석이 있다` 완곡.
- 인라인 출처 `(출처: URL)` — 공식 발표·원문 인용·최신성 주장만.
- 평범한 설명에는 출처 X. 글 끝 References X.

## 15. 카드 콘텐츠

카드 이미지는 본문 H2 메시지를 시각으로 한 번 더 전달. 양식보다 메시지 먼저.

- H2 제목이 아니라 그 섹션이 전하는 결론을 한 문장으로 적고, 그 문장에서 슬롯을 푼다.
- 자리채움 (`이미지: section-N`, 카테고리 이름만) 금지.
- 메시지가 양식과 안 맞으면 양식을 바꾼다 — 메시지를 비틀지 않는다.
- 양식별 적합 H2 메시지·시각·필드 명세는 `blog-cards/SKILL.md` 가 단일 출처.
- 한 글 안 카드 배경 톤은 모두 같은 그룹 (다크/라이트/컬러풀). 섞지 않는다.
- 카드 title·subtitle 도 본문 톤과 같은 자연 한국어.

## 16. X 아티클 요약 (SNS 톤)

블로그 본문은 평서체지만 X 요약은 SNS 톤.

- 첫 문장 후킹 — 호기심 자극, 구체 수치·갈등·결과.
- 핵심 변화 1~2 개와 체감 포인트 중심.
- 줄 바꿈 자유, 한 줄 후킹 질문 OK.
- 어색 패턴 (3 절 표) 은 SNS 톤에서도 회피.

## 17. 파일명

- `YYYY-MM-DD-<핵심 키워드 한글 슬러그>.md`
- 한글 그대로. 공백 → 하이픈. 마침표·괄호·물음표 제거.

## 18. 톤 — 읽기 쉬운 반구어체

평서체 기본을 유지하되 격식체보다 **읽기 쉬운 반구어체**.

- 비문·번역투·직역체는 자연스럽게 수정.
- 긴 문장 짧게. 한 문장 한 내용.
- 반복 표현·불필요한 설명은 줄인다.
- 설명조 (`~ 라는 점이다`, `~ 한다는 사실이다`) 보다 직접 서술 (`~ 다`).
