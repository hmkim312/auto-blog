# 블로그 글쓰기 규칙

자연스러운 한국어 기술 블로그 글을 쓰는 절차와 톤을 정의한다. 이 문서가 단일 정관이다 — 다른 SKILL.md·메모리는 보조 자료다.

## 1. 글쓰기 핵심 원칙

블로그를 쓰는 모든 단계가 다음 다섯 가지에서 출발한다.

1. **먼저 말한다** — 결론을 첫 문장에 둔다. 평론·격언으로 우회하지 않는다.
2. **짧게 쓴다** — 한 문장 40 자 이내, 한 단락 2~4 문장.
3. **능동형으로 쓴다** — `~ 됐다`, `~ 시켜진다`, `~ 가 되어 있다` 같은 수동형은 영어 직역인 경우가 많다. 능동형으로 다시 쓴다.
4. **한 문장에는 한 가지 내용만** — 영어식 긴 수식·관계절·콤마 나열은 단문 둘로 끊는다.
5. **추상 주장은 숫자·예시·코드·상황으로 받친다** — 한 줄로 마치고 다음 줄에 구체 사례.

## 2. 작업 절차 — 한국어 사실 카드 먼저

영문 자료를 LLM 한 번에 한국어 글로 옮기면 어순·동사 결합·명사화가 영어식으로 따라온다. 그래서 **번역과 작문을 분리한다**.

### 2-1. 한국어 사실 카드로 옮기기

영문 자료에서 사실·수치·인용만 뽑아 한국어 한 문장씩 카드로 정리한다.

```markdown
### 카드 1: 정의
- 사실: 하네스 엔지니어링은 모델 환경의 도구·권한·피드백 루프를 짜는 분야다.
- 시점: 2026년 3월부터 표현이 부쩍 자주 보였다.
- 출처: nxcode.io/...
```

규칙:
- 영어 단어·관용구 그대로 옮기지 않는다.
- 영어 수동태는 한국어 능동형으로 바꾼다 (`is reported` → `보고했다`).
- 영어식 명사구 (`reliability problems`) 는 동사형으로 풀어 쓴다 (`결과가 흔들리는 문제`).
- 의견·해석·맥락은 추가하지 않는다. 사실만.

### 2-2. 사실 카드 기반 한국어 작문

작성자는 한국어 사실 카드만 본다. 영문 원문은 노출하지 않는다.

- 카드 사실을 골든 톤 표본 결로 다시 짠다.
- 카드에 없는 도구·가드·단계는 끌어다 적지 않는다.
- 영어식 어순이 따라오는 표현이 보이면 즉시 다시 쓴다.

## 3. 자연스러운 대체 표현

어색한 표현이 보이면 다음 대체로 다시 쓴다. 문맥상 자연스러우면 무리하게 금지하지 않는다.

| 어색 | 자연스러운 대체 |
|------|----------------|
| 자리 (추상 의미) | 곳, 부분, 구간, 지점, 상황 |
| 박다 / 박혀 있다 / 박아 두다 | 적다, 기록하다, 저장하다, 들어 있다, 있다 |
| 잡다 / 잡히다 (검출 의미) | 찾다, 확인하다, 발견하다, 가려내다 |
| 모델 둘레 | 모델 환경, 모델 주변, 모델 바깥 |
| ~ 로 넘어간 ~ 문제 | ~ 만으로는 안 풀리는 ~, ~ 가 ~ 로 옮겨 갔다 |
| ~ 을 다루다 / ~ 을 다루는 분야 | ~ 에 집중한다, ~ 를 본다, ~ 를 푼다 |
| ~ 에 가깝다 (반복) | ~ 다, ~ 인 셈이다 |
| ~ 다는 뜻이다 (반복) | ~ 다, ~ 라는 의미다 |
| ~ 다는 분석이 늘었다 | ~ 다고 자주 짚힌다, ~ 가 흔히 보인다 |
| ~ 에 자리 잡았다 (반복) | ~ 가 됐다, ~ 가 자리매김했다 |
| ~ 처럼 읽혔다 / 들렸다 | ~ 라고 받아들였다 |
| ~ 기 때문이다 (남용) | ~ 다, 구체 이유로 |
| A 와 B 와 C | A·B·C, A 나 B, C |
| 한 겹 / 한 겹 더 | 한 단계, 한 층 |
| 결합 효과, 강점이 있다 | 함께 쓰면 좋다, 잘한다 |

## 4. 번역투 회피

영문 자료를 옮길 때 가장 흔한 직역 패턴 네 가지.

1. **수동태 → 능동형**
   - `결과가 흔들리게 된다` → `결과가 흔들린다`
   - `~ 가 적용되어 있다` → `~ 를 적용했다`
   - `보고됐다고 알려졌다` → `보고했다`, `짚었다`
2. **추상 명사 마무리 → 동사형**
   - `~ 한 흐름`, `~ 한 분야` 같은 추상 명사 마무리는 동사로 풀어 쓴다.
3. **콤마 나열 → 단문 분해**
   - 명사 5~6 개 콤마 결합 한 문장은 단문 둘로 끊는다.
4. **`~ 에 가깝다 / ~ 라는 뜻이다 / ~ 부분 / ~ 영역` 남용 — 한 글에 2 회 이하**

## 5. 골든 톤 표본 (조대협 블로그 기준)

추상 룰이 헷갈릴 때 다음 표본 결로 본다.

- **상황 제시 도입**: "Claude Code 로 복잡한 작업을 하다 보면 한 가지 문제를 마주치게 된다. 코드베이스를 탐색하고, 분석하고, 수정하고, 테스트까지 하다 보면 context window 가 빠르게 소모된다는 것이다."
- **결론 먼저**: "SubAgent 는 Claude Code 를 훨씬 더 강력하게 만드는 핵심 기능이다."
- **비유로 풀기**: "그렇다면 어떻게 하면 좋을까? 인간 세계에서는 이 문제를 이미 해결했다. 바로 전문화이다."

핵심: 동사형 종결 70% 이상, 영문 기술 용어 그대로, 강조어 (`바로`, `특히`, `핵심은`) 자유, 비유 권장.

## 6. 대상 독자

- AI 에 관심 있는 일반 독자 + 입문 개발자
- 전문 용어는 처음 등장할 때만 한 줄 풀이
- 비유·예시 적극 사용

## 7. 도입부

H1 → 썸네일 카드 → 두 단락 → 첫 H2 순서.

### 첫 단락 — 두 형태 중 골라 쓴다

**A. 정의형** (검색 키워드형 글)
- 100~160 자, 1~2 문장
- 핵심 키워드 자연 포함, 평서체, 1인칭 X

**B. 상황 제시형** (실무·문제 해결형 글, 조대협 톤)
- 100~180 자, 2~3 문장
- "~ 하다 보면 ~ 하게 된다" 식 상황 제시 + 한 줄 결론
- 핵심 키워드 자연 포함, 평서체, 1인칭 X

### 두 번째 단락 — 1인칭 후크

- 2~4 문장
- 본인 경험·동기·첫 인상 짧게
- 메타 멘트 (`이 글에서는 ~ 다룬다`) X
- 변명 (`안 써봐서 잘 모른다`) X

### 형식

- 썸네일 카드 마커는 H1 바로 아래
- 두 단락 사이 빈 줄 1 개
- 두 단락 다음 줄에 곧장 첫 H2

## 8. 제목

대제목·소제목은 그 섹션이 실제로 무엇을 말하는지 한 줄로 보이는 문장으로 짓는다.

### 종결 — 동사형 기본, 명사형도 OK

- 자연스러우면 동사형이 더 좋다.
- 명사형 종결을 강제하지 않는다.
- 어색한 명사형 (`~ 문제`, `~ 한 흐름`) 은 동사형으로 푼다.

### 정의형·의문형 허용

- `~ 란 무엇인가`, `왜 ~ 가 필요한가` 자유롭게 사용. 한 글에 4 개 이하.
- 의문문 H2 다음은 본문에서 그 질문에 직접 답한다.

### 금지 패턴

- `기술 설명`, `개발 경험 공유`, `오늘 배운 것` 그대로 X
- `~ 정리`, `~ 가이드`, `~ 한눈에 보기` 추상 마무리 한 글에 1 개 이하
- `세 가지 경로`, `네 가지 방법` 숫자 강박 한 글에 1 개 이하
- 영어 직역식 명사 묶음 (`사용 영역`, `권한 구조`) 풀어 쓴다

## 9. 글 구조

기본: **기술 설명 → [선택 섹션] → 개발 경험 공유 → 마치며**

선택 섹션은 주제에 맞을 때 1 개. 본인 경험이 없으면 경험 섹션 대신 "함정·실전 팁".

H2 직후에는 도입 단락 X. 곧장 H3 (또는 이미지 마커 → H3) 로 진입. 도입 단락은 H1 다음 두 단락에만.

### 섹션 1: 기술 설명
- 개념 → 왜 필요한가 → 어떻게 동작하나
- H3 3~4 개
- 코드 예시 1 개 이상 (코드가 의미 있을 때)

### 선택 섹션
- 왜 지금 이 주제인가 / 대안 비교 / 실전 팁·함정 피하기

### 섹션 2: 개발 경험 (또는 함정·실전)
- 본인 경험 있으면 1인칭
- 없으면 일반 함정·팁 (윤색 X)

### 마치며
- 자연스러운 문장 (명사형·동사형 모두 OK)
- 본문 3 줄 이내 요약
- "오늘 배운 것:" X
- "다음 글에서 다루겠다" X

## 10. 문체

- 평서체 (`~ 한다`, `~ 된다`, `~ 이다`, `~ 게 된다`) 가 기본
- 한 문장 40 자 이내 권장
- 어려운 단어 대신 쉬운 단어
- 이모지 X
- em-dash (—) X — 쉼표·마침표·괄호로 대체

### 자연스러운 강조어 — 허용

- `바로`, `특히`, `꼭`, `핵심은`, `중요한 것은` ← 한 단락 1~2 개
- `그렇다면`, `반대로`, `이제` ← 자유
- `마치 ~ 처럼`, `~ 같이` ← 비유 자유

### 금지 표현

- "첫째, 둘째, 셋째" 번호 나열
- "크게 N 가지다" 도식 강박
- "이를 통해", "따라서", "결론적으로" 남용 (한 글 1~2 개까지)
- 볼드 남용 (한 단락 1 개 이하)

## 11. 영어 용어 표기

기본: 한국 독자가 한글로 더 익숙한 단어만 음역. 영문 기술 용어는 그대로.

- **그대로**: SubAgent, MCP, Hook, context window, JSON, YAML, LangChain, API, GPU, CLI
- **한글 음역**: 클로드 코드, 클로드, 챗GPT, 제미나이 (한글 표기 통용)
- **첫 등장 병기**: 한글 음역하는 단어만 첫 등장에 한 번 영문 병기 (`클로드 디자인(Claude Design)`). 두 번째 이후 한글만.
- 코드·명령어·URL·파일명·태그명: 영문 그대로

## 12. SEO

글 1 개 = 핵심 키워드 1 개.

### 4 곳에 핵심 키워드 자연 포함
- H1 제목
- URL 슬러그·파일명 (한글 그대로, 공백 → 하이픈)
- 도입부 첫 단락
- 썸네일 alt

### 본문 자연 등장 5 회 이상

H2·H3 까지 강박적으로 넣지 않는다 (키워드 강박 X).

### 두 명사 조합 키워드

`클라우드 온프레미스` 같은 조합은 본문·제목에서 `와/과` 자연 결합 (`클라우드와 온프레미스`). 슬러그는 조사 빼고 (`클라우드-온프레미스`).

### H1 숫자 옵션

연도·N 가지·N 분이 자연스럽게 들어가면 클릭율↑. 어색하면 강제 X.

### 이미지 alt

- 본문 `![alt](path)` 의 alt 는 카드 내용을 한국어 30~80 자로 묘사
- 썸네일 alt 1 장에만 핵심 키워드 자연 포함
- 섹션 카드 alt 는 해당 H2 메시지 중심
- 같은 alt 두 번 이상 X

### 내부 링크 (Internal Link Juicer)

- 초안 작성 전 `internal-links.md` 확인
- 본문에서 ILJ 키워드 자연 등장 시 `<!--ILJ-->` 주석. 강제 X
- `## 인터럴 링크` 섹션 (`## 태그` 다음, 필수):
  - 나가는 링크: 표시한 키워드 + 연결 대상 글
  - 들어오는 링크: 다른 글이 이 글을 참조할 키워드 2 개
- 들어오는 링크 키워드는 `internal-links.md` 에도 추가

## 13. 카드 콘텐츠 작성 원칙

카드 이미지는 본문 H2 메시지를 시각으로 한 번 더 전달한다. 양식보다 메시지가 먼저.

### H2 메시지 한 문장으로

H2 제목이 아니라 그 섹션이 실제로 전하는 결론을 한 문장으로 적는다. 카드 슬롯은 그 문장에서 풀려 나온다.

### 슬롯 — 메시지 본질을 직접 적는다

좋은 짝 (`section-numbered`, H2: `입문할 때 자주 막히는 곳`):
- `이미지가 10GB 로 부풀어 오를 때`
- `latest 태그만 믿었다가 배포가 깨질 때`
- `캐시 레이어 이해하고 빌드 속도 잡기`

나쁜 짝 (같은 H2):
- `이미지 관리`
- `태그 관리`
- `빌드 관리`

자리채움 (카테고리 이름만, `이미지: section-N` 류) 금지.

### 양식에 맞춰 메시지 비틀기 X

메시지가 양식과 안 맞으면 양식을 바꾼다. 메시지를 비틀지 않는다.

### 양식별 잘 맞는 H2 메시지

| 양식 | H2 메시지 형태 |
|------|----------------|
| `thumbnail` | H1 의 정보형 한 줄 |
| `thumbnail-bold` | 한 줄로 결론을 강하게 드러내는 임팩트형 |
| `thumbnail-stat` | 핵심이 수치 1 개 |
| `thumbnail-quote` | 회고·1인칭 |
| `thumbnail-magazine` | 표지처럼 시각 임팩트 큰 글 |
| `section-hero` | H3 가 카드 한 장 분량 설명 N 개 |
| `section-simple` | H3 0~1 개, 결론 한 문장 |
| `section-numbered` | 순서·원칙·단계 명확한 H3 |
| `section-timeline` | 시간·버전·이전 → 이후 흐름 |
| `section-split` | 한 수치가 섹션 중심 |
| `section-stat-grid` | 수치 4 개 이상 |
| `section-compare` | H3 정확히 2 개 + 대립 |
| `section-callout` | 함정·경고·결론 단일 메시지 |
| `section-terminal` | 명령어·코드가 핵심 |
| `section-flow` | 파이프라인·노드 흐름 |

이 표가 양식 적합도의 단일 출처. `/blog-cards` SKILL.md 카탈로그는 시각·필드 명세.

### 카드 톤도 본문 톤과 정렬

카드 title·subtitle 도 본문 톤과 같은 자연 한국어. 슬로건·임팩트만 노린 추상 명사구 X.

## 14. 파일명 규칙

- `YYYY-MM-DD-<핵심 키워드 한글 슬러그>.md`
- 한글 그대로. 공백 → 하이픈. 마침표·괄호·물음표 제거.
- 예: `2026-04-19-도커-입문.md`

## 15. 분량

- 총 3,500 자 ±10% (3,150~3,850 자, 코드블록·`## 태그`·`## 이미지 메타` 제외)
