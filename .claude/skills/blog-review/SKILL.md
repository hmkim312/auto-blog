블로그 초안을 CLAUDE.md 규칙에 맞춰 검수하는 스킬. CLAUDE.md 가 단일 정관이고, 이 스킬은 그 룰을 결정론·LLM 판단으로 풀어 적용한다.

> **권장 모델:** Codex (GPT-5.5). Claude 자가 검수는 직번역체를 못 잡는 한계가 확인됐다 (2026-05-15). `/blog` 자동 루프에서는 Codex 자가 검수로 호출.

## 사용법

```
/blog-review                              # posts/ 최신 파일 검수
/blog-review posts/2026-04-22-docker.md   # 특정 파일 검수
/blog-review --fix                        # 최신 파일 검수 + 자동 교정
/blog-review <파일> --fix                 # 특정 파일 + 자동 교정
/blog-review <파일> --judge               # 자동 루프용 점수 리포트 출력
```

## `--judge` 모드 (자동 루프용)

`/blog` 스킬에서 호출하는 모드. JSON 점수 리포트만 출력해 호출자가 후속 판단에 쓰도록 한다.

**출력 포맷 (JSON):**

```json
{
  "score": 87,
  "verdict": "FAIL",
  "결정론_위반": [
    "L5: 음슴체 1건 ('~했음.')",
    "L17: em-dash 1건"
  ],
  "어색_패턴": [
    {"line": 24, "원문": "모델 둘레의 도구", "이유": "둘레=사물 외곽선, 추상에 안 붙음", "대안": "모델 환경의 도구"}
  ],
  "AI_톤": [
    "L31 '~ 다는 분석이 늘었다' — 영어식 직역 + 평론 톤"
  ],
  "후크_평가": "H1 정의형, 도입에 구체 수치·갈등 없음. 본문 H2-3 의 30위→5위 사례를 도입에 끌어 올릴 것 권장.",
  "추가_체크": {
    "SEO_4자리_일관": "OK",
    "인터럴_링크_마커": "OK",
    "인라인_출처": "OK"
  },
  "verdict_근거": "..."
}
```

**판정 기준:**
- `verdict`: `PASS` (score ≥ 90 + 결정론 위반 0 + 어색 패턴 0~1 + 추가 체크 모두 OK) / `FAIL` (그 외)
- 통과 컷오프 = **90점**

## 입력 해석

- 인자에 `.md` 파일 경로가 있으면 해당 파일
- 없으면 `posts/` 하위에서 수정 시각 가장 최근 `.md` 파일
- `--fix` 플래그가 있으면 자동 교정 단계까지 실행
- `--fix` 없으면 리포트만 (읽기 전용)

## 검수 항목

### A. 결정론 규칙 (정규식·카운팅)

위반 시 [FAIL], 통과 시 [OK].

1. **파일명 형식**: `^\d{4}-\d{2}-\d{2}-.+\.md$`
2. **H1 개수**: 정확히 1
3. **H2 개수·내용** (꼬리 메타 H2 `## 태그`, `## 인터럴 링크`, `## 이미지 메타`, `## X 아티클 요약` 은 카운트 제외):
   - 본문 H2 총 3 개(기본) 또는 4 개(선택 섹션 포함). 5 개까지 WARN
   - 본문 H2 에 "기술 설명", "개발 경험 공유", "오늘 배운 것" 그대로 들어가면 FAIL
4. **H3 개수**: 섹션당 3~4 개 권장 (2 개 이하 WARN)
5. **이모지 검출**: 본문 어디에도 이모지 X (유니코드 이모지 범위 탐지)
6. **분량** (마크다운 글자 수, frontmatter·코드블록·꼬리 메타·이미지 마커 제외):
   - 총 3,500 자 ±10% (3,150~3,850)
   - 섹션 1: 1,400 자 ±15%
   - 선택 섹션: 700 자 ±20%
   - 섹션 2: 1,100 자 ±15%
   - 마치며: 300 자 ±25%
   - 선택 섹션 없으면 섹션 1/2 각각 +300 자 허용
7. **문장 길이**: 40 자 초과 문장 개수 리포트 (3 개 이하 OK, 초과 시 WARN)
8. **문체**: 음슴체·비격식 검출
   - 금지 종결: `~했음\.`, `~하였음\.`, `~임\.`, `~함\.`, `~였음\.`
   - 위반 문장 최대 5 개 인용
9. **마치며 금지 문구**: "오늘 배운 것:" 그대로 들어가면 FAIL

9-b. **가독성 — 문단 나누기**:
   - H3 섹션 본문이 빈 줄 없이 한 덩어리면 FAIL
   - 한 문단 5 문장 초과 WARN
   - 한 H3 안에 문단 1 개뿐이면 WARN

9-c. **도입부 두 단락 (SEO description 자리)** — H1 직후 영역, 이미지·HTML 주석 제외 텍스트 단락만 카운트:
   - 단락 개수 2 개 아니면 FAIL
   - **첫 단락 길이 100~180 자** 벗어나면 WARN (정의형 100~160, 상황 제시형 100~180)
   - 첫 단락에 1인칭 어투 (`나는`, `내가`, `처음엔`, `~했었다`) 검출되면 FAIL — 정의 또는 상황 제시이지 1인칭 후크 아님
   - 두 단락 합산 250 자 미만이면 WARN
   - `## 도입부` 같은 H2 제목이 본문에 들어가 있으면 FAIL
   - 썸네일 마커가 H1 바로 아래에 없으면 FAIL

10. **콘텐츠 요소**:
   - 코드 블록 1 개 이상 (없으면 WARN)
   - 코드 블록 언어 명시 + 주석 1 줄 이상 (없으면 WARN)
   - 비교 표·체크리스트 — 권장 항목에서 제외 (CLAUDE.md 가독성 룰에 X)

10-b. **핵심 키워드 등장 횟수** (CLAUDE.md `## SEO`):
   - 본문 전체에서 핵심 키워드 5 회 미만 WARN
   - H1·도입부·썸네일 alt 3 자리 누락 시 각각 FAIL

10-c. **이미지 alt 텍스트** (CLAUDE.md `## SEO`):
   - alt 비어 있거나 `이미지: 썸네일` / `이미지: section-N` 자리채우기면 FAIL
   - alt 길이 30~80 자 범위 벗어나면 WARN
   - 같은 alt 두 번 이상 FAIL
   - em-dash 부제 alt FAIL
   - 본문 H2 직후 section-N 카드 마커 없으면 WARN

10-d. **em-dash (—) 사용 금지**:
   - 본문·제목 어디에서든 em-dash 검출 시 FAIL
   - 위반 시 `쉼표/마침표/괄호로 대체` 안내

10-e. **어색 패턴 grep 카탈로그** (NEW — CLAUDE.md `## 톤 가드` 의 누적 카탈로그를 정규식으로):

   본문에서 다음 패턴 검출 시 [FAIL] + 해당 줄 인용 + 대안 제시. 카탈로그는 누적되며 새 어색 자리가 발견되면 추가.

   ```
   "모델 둘레", "둘레의 ~", "둘레 환경"             → "환경", "주변", "바깥"
   "~로 넘어간 신뢰성", "~로 넘어간 ~ 문제"         → "~만으로는 안 잡히는"
   "~을 다루는 새 분야", "~을 다루며"               → "~에 집중하는", "~를 본다"
   "~다는 분석이 늘었다", "~다는 분석이다"          → "~다고 자주 짚힌다"
   "~에 자리 잡았다"                                 → "~가 됐다"
   "~ 신뢰성 문제", "~ 한 자리", "~ 한 흐름"        → 동사형으로
   "~ 처럼 읽혔다", "~ 처럼 들렸다"                  → "~ 라고 받아들였다"
   "~다는 뜻이다" (한 글에 2회 이상)                 → "~다", "~인 셈이다"
   "[가-힣]+와 [가-힣]+와 [가-힣]+"                  → "A·B·C" 또는 "A 나 B, C"
   "한 겹", "한 겹 더"                               → "한 단계", "한 층"
   "결합 효과", "한도에 닿다", "강점이 있다", "정가 옆에 우회로"
   ```

   각 패턴 정규식은 한국어 주변 어휘 결합까지 본다 (예: `둘레` 단독은 OK, `[가-힣]+ 둘레의` 결합 시 FAIL).

### B. LLM 판단 항목

리포트에 [판단] 섹션으로 정리.

11. **입문자 눈높이**: 풀이 없는 전문 용어 5 개까지 지적
12. **비유·예시 활용도**: 개념 설명에 비유 또는 예시 있는지
13. **섹션 흐름**:
   - 섹션 1: 개념→필요성→동작 순서인지
   - 선택 섹션: 주제와 맞나, 다른 섹션과 중복 없나
   - 섹션 2: 1인칭 또는 함정·실전 톤인지
   - 마치며: 3 줄 이내 요약인지

14. **제목·소제목 품질**:
   - 정의형·의문형 (`~ 란 무엇인가`, `왜 ~ 가 필요한가`) 은 **허용**. 한 글에 4 개 초과면 WARN
   - 각 H2/H3 가 그 섹션의 실제 내용을 드러내는 구체 명사·동사 포함하는지
   - 추상·템플릿이면 대안 제목 제안
   - **명사형 종결 강제 X** — 동사형이 자연스러우면 동사형 OK (CLAUDE.md `## 제목 구조` 절)

15. **전체 톤 일관성**: 음슴체 섞이지 않고 일관된지
16. **선택 섹션 적절성**: 풍부하게 하는지, 억지로 끼운 느낌인지

17. **자연 한국어 종합 판단** — LLM 종합. 결정론 grep 카탈로그가 못 잡는 결을 본다.
   - **영어 어순 직번역**: "한국어 글쓰기 책에서 본 적 없는 어순·구문". 어휘뿐 아니라 어순 단위까지. 어색하면 인용 + 자연 한국어 대안.
   - **명사화 마무리 어색**: `~ 부분`, `~ 자리`, `~ 영역` 반복 (한 글에 3 개 이하)
   - **격언·평론 톤**: 격언형 마무리 (`작은 규칙이지만 누적되면 결과가 달라진다`), 평론 (`~ 한 이름이다`) 지적
   - **메타 멘트**: `이 글에서는 ~ 다룬다`, `이 글의 줄거리다` 지적
   - **구어·은어**: `박힌다`, `때려넣`, `갈아탄다` 등
   - **풀이 없는 외래어** (일반 독자 대상 글 한정)
   - **CLAUDE.md `## 골든 톤 표본` 과 비교** — 조대협 톤과 결이 다른 자리 짚음

   각 위반은 문장 인용 + 대안 1 개. 5 건 이상이면 [FAIL] 강하게.

18. **키워드 강박 + 카드 정렬 + 경험 사실성**:
   - **키워드 강박**: 한 키워드가 H1·H2·H3·도입·결론·카드까지 다 박히면 본질 가려짐. H2·H3 는 본질·메커니즘으로 다시 쓰도록 제안.
   - **카드 ↔ 본문 정렬**: 카드 타이틀이 H2 메시지와 어긋나거나 라벨 의미 없으면 지적.
   - **카드 title/subtitle 문장 쪼개기 금지**: `title: "짧을수록"` + `subtitle: "잘 따른다"` 처럼 한 문장 분할 FAIL.
   - **경험 사실성**: 본인 답변에 없는 도구·가드 끌어다 적은 흔적 윤색이라 지적.
   - **18-a. H2-카드 적합도** (CLAUDE.md `## 카드 콘텐츠 작성 원칙`): 각 본문 H2 메시지가 해당 `section-N` 카드 슬롯에 직접 담겼나. 자리채움·양식 욱여넣음 짚음.
   - **18-b. `--fix` 교정 범위 한정**: 슬롯 텍스트만. palette/template/슬롯 개수 변경 X.
   - **18-c. 교정 diff 리포트**: 형식 `<카드 키>.<슬롯 경로>: "전" → "후"`

19. **유입 잠재력 점수 (100 점, 90+ PASS / 90 미만 FAIL)** — 5 축 합산:
   - **키워드 매칭** (20): H1·도입 첫 단락·썸네일 alt·파일명 슬러그 4 자리에 동일 키워드 자연 포함
   - **신선도** (20): 7 일 이내 사건 또는 영구 검색 수요
   - **제목 클릭율** (20): 구체 수치·인물·결과·갈등 박혔나
   - **차별화** (20): 본인 시각·경험 박혔나
   - **SEO 첫 단락** (20): 100~180 자, 정의 또는 상황 제시, 1인칭 X

   출력 형식:
   ```
   유입 점수: 92 / 100 (PASS)
     - 키워드 매칭: 18/20
     - 신선도: 16/20
     - 제목 클릭율: 19/20
     - 차별화: 20/20
     - SEO 첫 단락: 19/20
   ```

   90 미만이면 가장 낮은 축 1 개 짚어 보강 제안 한 줄.

## 출력 포맷

```
# 블로그 리뷰 리포트
파일: posts/2026-05-15-하네스-엔지니어링.md
총평: PASS / FAIL (결정론 위반 N 개, 유입 점수 N/100)

## 결정론 체크
[OK]   파일명 형식
[FAIL] 어색 패턴 grep — L24 "모델 둘레의 도구"
[WARN] 40 자 초과 문장: 3 개

## 분량
총 3,665 자 (기준 3,150~3,850 → OK)

## 판단 (LLM)
- 자연 한국어: L31 "~ 다는 분석이 늘었다" 직번역
- 후크: H1 정의형, 30 위→5 위 수치를 도입에 끌어 올릴 것
...

## 교정 제안
1. L24 "둘레의" → "환경의"
2. L31 "분석이 늘었다" → "자주 짚힌다"
```

## 자동 교정 (`--fix`)

**수정 대상 (안전한 치환)**:
- 음슴체 → 평서체 (`~했음.` → `~했다.` 등, 코드블록 제외)
- 이모지 제거
- 마지막 H2 가 마무리 화이트리스트 외면 `## 마치며` 로 교체
- "오늘 배운 것:" 제거
- em-dash (—) → 쉼표·마침표·괄호
- 어색 패턴 grep 카탈로그 1:1 치환 가능 자리 (예: "둘레의" → "환경의")

**수정 금지 (LLM 판단 영역)**:
- 분량 조정 (제안만)
- 제목·소제목 변경
- 문장 구조 변경
- 전문 용어 풀이 추가

교정 후 diff 출력 + 원본은 `<파일>.bak` 백업.

## 실행 흐름

1. 대상 파일 결정
2. 결정론 체크 A 수행
3. LLM 판단 B 수행 (CLAUDE.md `## 골든 톤 표본` 과 비교)
4. 리포트 출력
5. `--fix` 면 안전 치환 + 백업 + diff
6. 종료

## 규칙

- CLAUDE.md 가 단일 정관. 충돌 시 CLAUDE.md 우선.
- frontmatter·코드블록·꼬리 메타 섹션 (`## 태그` 부터 끝까지) 은 분량·문체 체크 제외
- 본문 이미지 마커도 분량 계산 제외
- `--fix` 는 반드시 백업 후 수정
- 어색 패턴 카탈로그는 누적. 새 자리 발견되면 10-e 에 정규식 추가.
