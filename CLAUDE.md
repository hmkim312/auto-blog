# 블로그 글쓰기 규칙

## 대상 독자
- AI에 관심 있는 모든 사람, 일반인
- 전문 용어는 반드시 한 줄 설명 추가
- 비유와 예시를 적극 활용

## 제목 구조

대제목·소제목은 **그 섹션에서 실제로 무슨 이야기를 하는지 한 줄로 보이는 문장**으로 짓는다. 독자가 제목만 훑어도 글의 흐름이 이해돼야 한다.

- 대제목(H1): 주제를 한 문장으로. "~란 무엇인가" 식 사전적 정의 제목은 피한다.
- 소제목(H2): 그 섹션의 결론·이야기 축을 직접 드러내는 문장.
- 소소제목(H3): 섹션 내 세부 단락. 구체적인 명사·동사 포함.

### 금지 패턴 (AI 템플릿 냄새)
- `~가 뭔가`, `~는 무엇인가`, `~란 무엇인가` 같은 정의형 공식
- `왜 ~인가`, `왜 써야 하나` 같은 내용 없는 질문 공식 (단, 그 섹션이 실제로 "왜"에 답한다면 허용)
- `어떻게 동작하나`, `어떻게 쓰나` 같은 추상적 공식
- `기술 설명`, `개발 경험 공유`, `오늘 배운 것` 문구를 제목에 그대로 쓰지 않는다
- 같은 글 안에서 질문형 제목을 3개 이상 반복하지 않는다 (템플릿감 유발)

### 좋은 제목 기준
- 명사 하나로 끝내지 말고 **동사·비유·결과**를 한 조각 얹는다
- 독자가 "어 이건 내 얘기네" 하고 걸려들 만한 구체 장면
- 제목만 모아 읽어도 글의 줄거리가 이어진다

### 예시 (개선 방향)

Before (템플릿):
```
# Docker란 무엇인가
## 컨테이너는 왜 등장했나
### 컨테이너가 뭔가
### 왜 써야 하나
```

After (내용 직접):
```
# 내 노트북에선 되는데 서버에선 안 될 때 — Docker 이야기

## 컨테이너는 결국 "가벼운 상자"다
### 프로세스 하나를 격리해서 들고 다닌다
### VM 과 뭐가 다른가 — 무게 차이
### docker run 한 줄이면 환경이 따라온다

## 나는 Docker 를 이렇게 망했다
### 이미지가 10GB 까지 부풀었던 날
### latest 태그만 믿다가 배포가 깨진 순간
### 캐시 레이어를 이해하고 나서 속도가 바뀌었다

## 마치며
```

"VM 과 뭐가 다른가" 처럼 질문형이 **자연스럽고 구체적**이면 써도 된다. 문제는 공허한 질문 공식(`~가 뭔가`)과 반복이다.

## 글 구조

기본 구조: **기술 설명 → [선택 섹션] → 개발 경험 공유 → 마치며**

선택 섹션은 주제에 맞을 때만 1개 추가한다. 기본 2섹션만으로 충분하면 넣지 않는다.

### 섹션 1: 기술 설명
- 개념 → 왜 필요한가 → 어떻게 동작하나 순서
- 소소제목(H3) 3~4개
- 코드 예시 1개 이상 (코드가 의미 있는 주제면 필수), 주석 포함
- 소제목은 위의 "제목 구조" 규칙을 따른다. 질문형 템플릿 반복 금지.

### 선택 섹션 (주제에 맞으면 하나 선택)
- **왜 지금 이 주제인가** — 배경, 트렌드, 실무에서 왜 중요해졌는가
- **대안 비교** — 경쟁 기술/방식과 어떻게 다른가
- **실전 팁 / 함정 피하기** — 흔히 빠지는 함정

### 섹션 2: 개발 경험 공유
- 1인칭으로 작성 ("나는 ~했다", "처음엔 ~인 줄 알았는데")
- 소소제목(H3) 3~4개
- 막혔던 부분, 삽질한 것, 깨달은 것 포함
- 독자가 같은 실수를 피할 수 있게

### 마치며
- 3줄 이내 요약
- 요약 형식으로 작성, "오늘 배운 것:" 문구는 쓰지 않는다
- 다음 단계 한 가지 제안

## 문체
- 평서체 (~했다, ~이다, ~한다, ~였다)
- 짧은 문장 (한 문장 40자 이내 권장)
- 어려운 단어 대신 쉬운 단어
- 이모지 사용 안 함

## 가독성 — 문단 나누기 (필수)

블로그는 화면에서 읽히므로 긴 덩어리 글은 반드시 쪼갠다.

- **한 문단 2~4문장**을 원칙. 5문장 넘어가면 끊을 지점을 찾는다.
- **문단과 문단 사이는 빈 줄(블랭크 라인) 1줄**로 구분한다. 마크다운에서 줄바꿈만으로는 문단이 나뉘지 않으므로, 에디터에서는 **더블 엔터**로 친다.
- **주제·시점·어조가 바뀌는 지점**에서 무조건 끊는다. 예: 개념 설명 → 예시, 사실 나열 → 해석, 외부 인용 → 내 의견.
- 수치·인용·예시가 이어지는 경우, 각 항목마다 단락을 분리해도 된다. 나열이 길어질수록 쪼갠다.
- 한 섹션(H3) 안에 문단이 2~4개 나오는 게 기본. 하나의 긴 덩어리로 쓰지 않는다.

### AI 글 같은 표현 금지
- "첫째, 둘째, 셋째" 나열 구조 금지
- "크게 N가지다", "크게 나누면" 식 도입 금지
- "이를 통해", "따라서", "결론적으로", "핵심은", "중요한 것은" 금지
- **볼드** 남용 금지 — 한 문단에 1개 이하
- 목록(\-) 나열보다 자연스러운 문장 흐름으로 연결

### 검색 결과 인용 규칙
- 검색으로 가져온 정보를 확인 없이 단정적 사실로 쓰지 않는다
- 확인되지 않은 수치·주장은 반드시 출처 표기 또는 제외
- 커뮤니티 반응·사용자 의견은 "~라는 반응이 있다", "~로 알려졌다" 식으로 표현
- AI가 알고 있는 것도 검색 결과와 다를 수 있다. 검색 결과와 충돌하면 검색 결과 우선

## 파일명 규칙
- 형식: `YYYY-MM-DD-주제-슬러그.md`
- 예: `2026-04-19-docker-입문.md`

## 분량
- 총 3,500자 ±10% (3,150~3,850자, 마크다운 기준, 코드블록·이미지 프롬프트 제외)
- 섹션 1(기술): 1,400자
- 선택 섹션(있을 때): 700자
- 섹션 2(경험): 1,100자
- 마치며: 300자
- 선택 섹션이 없으면 섹션 1/2가 각각 +300자씩 늘어나도 됨
- 분량을 채우려 빈 말·중복·AI 투로 늘리지 않는다. 근거 없이 길어질 바에 짧게 마친다. 길이는 "검색 근거·경험·예시가 실제로 있을 때"만 늘어난다.
