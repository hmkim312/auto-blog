블로그 초안을 CLAUDE.md 규칙에 맞춰 검수하는 스킬.

> **권장 모델:** Sonnet 4.6. 결정론 체크 + LLM 판단 모두 Sonnet으로 충분. Opus면 `/model sonnet` 으로 전환 권장.

## 사용법

```
/blog-review                              # posts/ 최신 파일 검수
/blog-review posts/2026-04-22-docker.md   # 특정 파일 검수
/blog-review --fix                        # 최신 파일 검수 + 자동 교정
/blog-review <파일> --fix                 # 특정 파일 + 자동 교정
```

## 입력 해석

- 인자에 `.md` 파일 경로가 있으면 해당 파일
- 없으면 `posts/` 하위에서 수정 시각 가장 최근 `.md` 파일
- `--fix` 플래그가 있으면 자동 교정 단계까지 실행
- `--fix` 없으면 리포트만 출력 (읽기 전용)

## 검수 항목

### A. 결정론적 규칙 (정규식/카운팅으로 체크)

모두 **위반 시 리포트에 [FAIL]** 표시, 통과 시 [OK].

1. **파일명 형식**: `^\d{4}-\d{2}-\d{2}-.+\.md$` 여부
2. **H1 개수**: 정확히 1개
3. **H2 개수/내용** (꼬리 메타 H2 `## 태그`, `## 이미지 메타` 는 H2 카운트에서 **제외**):
   - 본문 H2 총 3개(기본) 또는 4개(선택 섹션 포함). 기술설명이 2개로 쪼개지면 5개까지 허용 (WARN)
   - 본문 마지막 H2 = **CLAUDE.md `## 글 구조 → 마무리` 의 화이트리스트 8개 중 하나**. 그 외면 FAIL.
   - 본문 H2에 "기술 설명", "개발 경험 공유", "오늘 배운 것" 문구가 **그대로** 들어가면 FAIL
4. **H3 개수**:
   - 섹션 1, 섹션 2 각각 3~4개 권장 (2개 이하면 WARN)
   - 선택 섹션은 2~3개 권장
5. **이모지 검출**: 본문 어디에도 이모지 없어야 함 (유니코드 이모지 범위로 탐지)
6. **분량** (마크다운 글자 수, frontmatter·코드블록·꼬리 메타 섹션 제외):
   꼬리 메타 = `## 태그` 부터 파일 끝까지 (`## 태그` → `## 이미지 메타`). 본문 내 `![이미지:…](…)` 마커도 길이 계산에서 제외.
   - 총 3,500자 ±10% (3,150~3,850자)
   - 섹션 1(기술): 1,400자 ±15%
   - 선택 섹션(있을 때): 700자 ±20%
   - 섹션 2(경험): 1,100자 ±15%
   - 마치며: 300자 ±25%
   - 선택 섹션이 없으면 섹션 1/2 각각 +300자 허용
7. **문장 길이**: 40자 초과 문장 개수 리포트 (3개 이하면 OK, 초과 시 WARN)
8. **문체**: 본문 음슴체/비격식 문장 패턴 검출
   - 금지 종결: `~했음\.`, `~하였음\.`, `~임\.`, `~함\.`, `~였음\.`
   - 허용 종결: `~했다`, `~이다`, `~한다`, `~였다`, 의문형(`~인가`, `~나?`), 명령/권유
   - 위반 문장 있으면 해당 문장 최대 5개 인용해 리포트
9. **마치며 금지 문구**: "오늘 배운 것:" 그대로 들어가면 FAIL
9-b. **가독성 — 문단 나누기**:
    - 각 H3 섹션 본문이 빈 줄 없이 한 덩어리로 이어지면 FAIL
    - 한 문단이 5문장 초과하면 WARN (끊을 지점 제안)
    - 한 H3 안에 문단이 1개뿐이면 WARN (2~4개 권장)
9-c. **도입부 두 단락 (SEO description 자리)** — H1 직후 영역에서 이미지·HTML 주석 줄을 제외한 텍스트 단락만 카운트:
    - 단락 개수가 2개가 아니면 FAIL (1개면 SEO 한 줄이 빠진 것, 3개 이상이면 description 가 잘려 깨짐)
    - **첫 단락 길이 100~160자** 범위를 벗어나면 WARN (구글 description 권장 길이)
    - 첫 단락에 1인칭 어투(`나는`, `내가`, `처음엔`, `~했었다` 등) 검출되면 FAIL — 정의문이어야 함
    - 두 단락 합산 250자 미만이면 WARN (description 가 H2 까지 끌고 갈 위험)
    - `## 도입부` 같은 H2 제목이 본문에 들어가 있으면 FAIL (룰 문서 안 구분 이름이지 본문 H2 가 아님)
    - **썸네일 마커**(`![<alt>](...)`) 가 H1 바로 아래에 없으면 FAIL — CLAUDE.md `## 도입부` 절 형식: H1 → 썸네일 → 두 단락 → 첫 H2
    - 위반 시 리포트 예: `[FAIL] 도입부 단락 1개 — SEO 정의 한 줄 누락. 썸네일 바로 아래에 100~160자 정의문 추가 필요.`
10. **콘텐츠 요소 포함 여부**:
    - 코드 블록 1개 이상 존재 (없으면 WARN)
    - 코드 블록에 언어 명시 및 주석 1줄 이상 (없으면 WARN)
    - 비교 표(`| --- |` 패턴) 1개 이상 권장 (없으면 INFO)
    - 체크리스트(`- [ ]` 또는 `- [x]`) 또는 요약 블록(`>`) 1개 이상 권장 (없으면 INFO)

10-b. **핵심 키워드 등장 횟수 (CLAUDE.md `## SEO` 절)**:
    - 파일명 슬러그 기준 핵심 키워드를 본문 전체(마크다운 alt·코드블록 제외)에서 카운트. 5회 미만이면 WARN.
    - H1·도입부·썸네일 alt 3자리 누락 시 각각 FAIL.

10-c. **이미지 alt 텍스트 (CLAUDE.md `## SEO` 절)**:
    - 본문 모든 `![alt](path)` 의 alt 자리가 비어 있거나 `이미지: 썸네일` / `이미지: section-N` 같은 자리채우기면 FAIL.
    - alt 길이 30~80자 범위 벗어나면 WARN.
    - 같은 alt 가 두 번 이상 나오면 FAIL (카드별로 다른 묘사).
    - `![제목 — 부제](...)` 형식처럼 em dash(`—`) 뒤에 부제를 붙인 alt 는 FAIL. alt 는 H1 제목 그대로 또는 짧은 단일 묘사만 허용.
    - 본문 H2 직후에 해당 섹션 카드 이미지(`section-N.webp`)가 없으면 WARN.
    - 위반 시 해당 줄 인용.

### B. LLM 판단 항목 (결정론으로 어려운 것)

리포트에 **[판단]** 섹션으로 정리.

11. **입문자 눈높이**: 전문 용어에 한 줄 설명이 붙어 있는지. 설명 없이 튀어나오는 용어 5개까지 지적.
12. **비유/예시 활용도**: 개념 설명에 비유 또는 예시가 있는지. 없으면 어느 부분에 넣으면 좋을지 제안.
13. **섹션 흐름**:
    - 섹션 1: 개념→필요성→동작 순서인지
    - 선택 섹션(있을 때): 주제에 맞는 방향(배경/비교/팁)인지, 다른 섹션과 내용 중복 없는지
    - 섹션 2: 1인칭인지, 삽질/깨달음 포함인지
    - 마치며: 3줄 이내 요약 + 다음 단계 1가지인지
14. **제목·소제목 품질**:
    - 금지 패턴 검출: `~가 뭔가`, `~는 무엇인가`, `~란 무엇인가`, `왜 써야 하나`, `어떻게 동작하나`, `어떻게 쓰나` 같은 공허한 공식 (해당 섹션이 실제로 구체적 답을 담고 있으면 허용)
    - 질문형 제목이 한 글 안에 3개 이상 반복되면 WARN
    - 각 H2/H3 가 "그 섹션의 실제 내용"을 드러내는 구체 명사·동사를 포함하는지. 추상적·템플릿 느낌이면 대안 제목 제안
15. **전체 톤 일관성**: 음슴체가 섞이지 않고 일관된지
16. **선택 섹션 적절성**: 선택 섹션이 있다면 실제로 내용을 풍부하게 하는지, 아니면 억지로 끼워 넣은 느낌인지 판단

17. **전문가 톤 + 자연스러운 한국어** — LLM 이 종합 판단한다 (정규식 X). 4가지 결을 모두 본다.
    - **비속어·구어**: 일반 독자 대상 정보글에 안 어울리는 비격식·은어. 대표 예: `박힌다`, `때려넣`, `지멋대로`, `자기 식대로`, `갈아탄다`. 한 건이라도 보이면 인용 + 대안.
    - **영어 직번역 어색함**: "한국어 글쓰기 책에서 본 적 없는 명사 묶음·동사 결합"이 핵심 판단. 어휘 단위(예: drop→흘린다)뿐 아니라 어순·구문 단위까지 본다. 어색하면 "이건 영어를 한국어로 억지로 옮긴 표현이다" 라고 지적하고 자연스러운 한국어로 다시 쓴다.
    - **에세이톤·격언**: 정보 전달 블로그에서 명상조 회상(`한참 멍했다`)·격언형 마무리(`~ 한 칸이 9초를 막아 준다`)·1인칭 감정 표현이 본문(기술/비교) 안에 섞이면 지적. 1인칭은 도입부 두 번째 단락 + 경험 섹션에만 허용.
    - **풀이 없는 외래어/약어**: IT 용어가 일반 독자 대상 글에 풀이 없이 등장하면 지적 (예: 토큰 → "운영 환경 접근 키").

    각 위반은 문장 인용 + 대안 1개. 한 글에 5건 이상이거나 글 전반에 깔려 있으면 [FAIL] 강하게 보고.

18. **키워드 강박 + 카드 정렬 + 경험 사실성** — LLM 판단.
    - 키워드 강박: 한 핵심 키워드(특정 수치·단어)가 H1·H2·H3·도입·결론·카드까지 다 박히면 본질이 가려졌다고 본다. H2·H3·본문 카드 타이틀은 본질·메커니즘·교훈으로 다시 쓰도록 제안.
    - 카드 ↔ 본문 정렬: 카드 타이틀이 본문 H2 의 메시지와 어긋나거나, 카드 라벨이 의미 없으면 (비대칭 라벨·빈 라벨) 지적.
    - 카드 title/subtitle 문장 쪼개기 금지: `title: "짧을수록"` + `subtitle: "잘 따른다"` 처럼 한 문장을 앞뒤로 나눈 형태는 FAIL. title과 subtitle은 각각 독립적 의미를 가져야 한다.
    - 경험 사실성: 본인 경험 진술이 사용자 답변(인터뷰)에 없는 도구·가드·단계를 끌어다 적은 흔적이 있으면 윤색이라 지적.

19. **유입 잠재력 점수 (100점, 90+ PASS / 90 미만 FAIL)** — 한국 독자가 검색·SNS 로 들어올 잠재력을 5축 합산으로 점수화한다. **결정론 위반과 동급으로 총평에 반영** — 점수 90 미만이면 결정론이 PASS 여도 총평을 FAIL 로 표기한다.

    5축 (각 20점):

    - **키워드 매칭** — 제목·도입 첫 단락·썸네일 alt·파일명 슬러그 4자리에 동일한 한글 핵심 키워드가 자연스럽게 들어가 있나 (CLAUDE.md `## SEO` 절). 4자리 모두 일관 18~20. H1 에 정의형/추상만 있으면 10점 이하. H2·H3·본문 카드까지 같은 키워드를 박았으면 키워드 강박이라 14점 이하.
    - **신선도/시의성** — 7일 이내 출시·발표·인수 사건과 결합되거나, 영구 검색 수요(요금제 비교, 입문 가이드)가 강한가. 둘 다 약하면 10점 이하.
    - **제목 클릭율** — 제목만 보고 클릭하고 싶은가. 구체 수치·인물·결과·갈등 조각이 들어가 있어야 18~20 (예: "이틀 만에 한도가 다 찼다"). 결론형이지만 후크 약하면 14~17. 정의형/템플릿이면 9 이하.
    - **차별화/1인칭** — 같은 주제 경쟁 글이 많을 때 본인만의 시각·경험이 보이는가. 본인 삽질·해석이 본문에 박혀 있으면 18~20. 단순 정리만이면 13 이하.
    - **SEO 첫 단락** — 도입 첫 단락(100~160자)이 정의 + 시점 + 핵심 사실을 담아 구글 description 자리에 그대로 잡힐 수 있는가. 1인칭이 섞이거나 도입부가 아예 없으면 9 이하.

    출력 형식:
    ```
    유입 점수: 92 / 100 (PASS)
      - 키워드 매칭: 18/20
      - 신선도: 16/20
      - 제목 클릭율: 19/20
      - 차별화: 20/20
      - SEO 첫 단락: 19/20
    ```

    90 미만이면 5축 중 가장 낮은 축 1개를 짚어 보강 제안 한 줄 추가.

```
※ 결정론 위반 0개여도 유입 점수 90 미만이면 총평 FAIL.

## 결정론 체크
[OK]   파일명 형식
[FAIL] H2 개수: 4개 (3개여야 함)
[WARN] 40자 초과 문장: 5개
[FAIL] 문체 위반 3건
   L24: "Docker는 컨테이너 기술임."
   L31: "나는 처음에 당황했음."
   ...

## 분량
총 1,820자 (기준 1,500 ±10% → OVER)
- 섹션 1: 850자 (기준 700 ±15% → OVER)

- 마치며: 250자 (OK)

## 판단 (LLM)
- 전문 용어: "네임스페이스", "cgroup" 설명 없이 등장
- 비유: 섹션 1에 비유 없음. "컨테이너 = 격리된 방" 정도 추가 권장
- 소제목 질문형: 섹션 1 H3 2개가 서술형

## 교정 제안
1. H2 "오늘 배운 것" → "마치며"로 변경 (L60)
2. 음슴체 3건 → 평서체 치환
3. 섹션 1 분량 150자 감축
```

## 자동 교정 (`--fix`)

**수정 대상 (안전한 치환만):**
- 음슴체 → 평서체 치환 (결정론 규칙 7번 위반): `~했음/~하였음/~임/~함/~였음.` → `~했다/~하였다/~이다/~한다/~였다.` (코드블록 제외)
- 이모지 제거
- 마지막 H2 가 CLAUDE.md 마무리 화이트리스트 외이면 `## 마치며` 로 교체
- "오늘 배운 것:" 문구 제거

**수정 금지 (LLM 판단 영역 / 위험):**
- 분량 조정 (자동 축약하지 않음, 제안만)
- 제목/소제목 변경 (의미 바뀔 수 있음)
- 문장 구조 변경
- 전문 용어 설명 추가

교정 후 diff 형태로 변경 내역 출력하고, 원본은 `<파일>.bak`으로 백업.

## 실행 흐름

1. 대상 파일 결정
2. 파일 읽어서 결정론 체크 A 수행 → 결과 수집
3. 본문을 LLM에 넣어 판단 항목 B 수행 → 결과 수집
4. 리포트 출력
5. `--fix`면 안전한 치환 수행 + 백업 생성 + diff 출력
6. 종료

## 규칙

- CLAUDE.md가 진리의 원천. 규칙 충돌 시 CLAUDE.md 우선.
- frontmatter(`---` 블록)와 코드블록은 분량 계산·문체 체크에서 제외.
- 꼬리 메타 섹션(`## 태그` → `## 이미지 메타`, 파일 순서)은 분량·문체 체크에서 모두 제외.
- 본문 이미지 마커(`![이미지: …](images/…/*.webp)`) 도 분량 계산 제외.
- `--fix`는 반드시 백업 후 수정.
