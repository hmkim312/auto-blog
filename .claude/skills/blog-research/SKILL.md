블로그 초안을 "주제 결정 → AI 쿼리 제안 → 사용자 승인 → Tavily 검색 → 심화(선택) → 검색 맞춤 Q&A → 초안" 루프로 만드는 스킬. 주제 인자가 있으면 그대로 사용, 없으면 8개 트렌드 소스에서 후보를 모아 사용자가 고른다. Q&A는 검색 결과에 따라 카테고리·질문이 동적으로 짜이고, 검색 스킵 시에만 고정 9문 Q&A 로 fallback.

> **권장 모델:** Sonnet 4.6. 글쓰기 품질은 Opus와 큰 차이 없고 비용·속도가 유리함. 현재 모델이 Opus면 `/model sonnet` 으로 전환 후 진행 권장.

## 사용법

```
/blog-research                              # 주제 탐색부터 시작
/blog-research uv 패키지 매니저              # 주제 직접 지정 → 0단계 스킵
/blog-research --profile hmkim              # 프로파일 적용 + 주제 탐색
/blog-research uv --profile hmkim           # 둘 다
```

## 실행 흐름 (사용자 승인 게이트 고정)

### 0단계: 주제 결정

**인자에 주제가 있으면 이 단계 전체를 스킵하고 1단계로 간다.** 사용자가 직접 입력한 주제 키워드를 그대로 쓴다.

주제 인자가 없을 때만 `scripts/discover-topics.py` 를 호출해 8개 소스(HN, GitHub Trending, arXiv cs.AI, OpenAI/Anthropic/DeepMind 블로그, GeekNews) 에서 최근 항목을 모은다. 무인증·무료 API 만 쓴다.

```bash
uv run scripts/discover-topics.py
```

stdout 의 JSON 배열을 받아 LLM이 다음 기준으로 점수화·중복 병합한 뒤 후보 10개를 **제목만** 번호 리스트로 출력한다 (출처·URL·근거 표시 금지):

1. **신선도** — 최근 7일 내 항목 가산. 30일 초과 감점.
2. **입문자 적합성** — 일반인이 들어도 호기심 가질 만한가? 학술 논문 raw 면 감점.
3. **글로 풀 만한 거리** — 블로그 한 편(3,500자) 분량 줄거리가 그려지는가?
4. **한국 검색 수요** — GeekNews 동시 등장 시 가산점.
5. **프로파일 적합성** — 프로파일이 있으면 `interests`/`niche` 와 겹치는지.

후보 출력 형식 (제목만, 그대로 `/blog-research` 다음 단계로 이어질 형태로 정제):

```
주제 후보 10개입니다. 번호로 고르세요. (다른 후보 보고 싶으면 "다시")

1. ...
2. ...
...
10. ...
```

**사용자 선택 처리**:
- 번호 → 해당 제목을 주제 키워드로 정제(필요 시 한국어 톤으로 한 줄 다듬어 사용자 확인). 다음 단계로 진행.
- `다시` / `다른 거` → 11~20위 후보로 다시 10개 출력. 한 세션 최대 2회.
- 자유 키워드 → 후보 안에서 가까운 것 1~3개 재추천. 또는 사용자가 직접 입력한 키워드를 주제로 채택.
- 모든 후보 거부 + 직접 입력도 없으면 종료.

### 1단계: 세션 ID 생성

Bash 툴로 `echo "blog-$(date +%Y%m%d-%H%M%S)-$$"` 를 실행해 세션 ID를 만든다. 이후 모든 `scripts/tavily-search.py` 호출을 `BLOG_RESEARCH_SESSION_ID=<이_ID> uv run ...` 형태로 전달한다. 스크립트는 이 세션 단위로 Tavily 호출을 최대 8회까지만 허용하고 9회째에 exit 4 로 거부한다.

### 2단계: 프로파일 로드

- 인자에 `--profile <name>` 이 있으면 `profiles/<name>.yml` 을 Read 로 읽는다.
- 파일이 없거나 인자가 없으면 기본값으로 진행: `niche` 빈 값, `reader_level` = `"AI 에 관심 있는 모든 일반인"` (CLAUDE.md 기본).
- 로드한 프로파일 요약을 사용자에게 3~4줄로 출력한다 (어떤 컨셉으로 진행하는지 확인).
- `search_bias.language` / `search_bias.region` 값이 있으면 6단계 Tavily 호출 시 `--language` / `--region` 로 그대로 전달한다.

### 3단계: Tavily 키 확인

`.env` 에 `TAVILY_API_KEY` 가 있는지 확인. 없으면 "토큰이 없다. `.env` 에 `TAVILY_API_KEY=` 를 추가하고 다시 시도하라" 를 안내하고 중단한다. (스크립트도 자체 검증하지만, 키 없으면 초기에 끊어 사용자 시간을 아낀다.)

### 4단계: AI 쿼리 제안

주제와 프로파일을 토대로 **5~7개의 Tavily 검색 쿼리 후보**를 번호 리스트로 제시한다. 다양성을 위해 서로 다른 각도(정의/트렌드/비교/실사용/성능/함정)로 섞는다.

출력 예:
```
주제: uv 패키지 매니저

검색 쿼리 후보 (5~7개 중 원하는 만큼 고르세요, 기본 권장 4~6개):
1. "uv python package manager 2026"       ← 최신 버전/기능
2. "uv vs poetry vs pip-tools comparison" ← 대안 비교
3. "uv PEP 723 inline scripts tutorial"   ← 실전 사용
4. "uv performance benchmark"              ← 성능 데이터
5. "uv common pitfalls issues"             ← 흔한 함정
6. "astral-sh uv roadmap"                  ← 향후 방향

번호로 선택하거나 편집/추가해 주세요. (예: "1,3,4,5" 또는 "1만 쓰되 뒤에 '2026' 붙여줘")
```

### 5단계: 사용자 쿼리 확정

사용자 응답을 받아 최종 쿼리 목록(**최대 6개**, 심화 검색 2회 여유 고려)을 에코한다. 0개 선택 시 6단계를 스킵하고 8단계로 간다 (검색 생략 경로).

### 6단계: Tavily 호출

확정된 각 쿼리를 **순차**로 호출한다. Bash 툴 사용:

```bash
BLOG_RESEARCH_SESSION_ID=<세션ID> uv run scripts/tavily-search.py \
  --query "<쿼리>" \
  --max-results 10 \
  --include-answer \
  [--language ko --region KR]
```

`--language`/`--region` 은 프로파일의 `search_bias` 가 있을 때만 붙인다. stdout 으로 나온 JSON 을 파싱해 각 결과를 "제목 — 출처 — 핵심 한 줄" 로 요약 출력한다. `answer` 필드가 있으면 "Tavily 요약" 으로 한 블록에 덧붙인다.

### 7단계: 심화 질문

"지금까지 본 결과 중 더 파고 싶은 키워드가 있나?" 라고 묻는다. 응답이 "없음"/"스킵"이면 8단계로 진행. 키워드를 받으면 8단계에서 추가 검색 **최대 2회**까지 수행한다.

### 8단계: 심화 검색 (선택)

심화 키워드가 있으면 Tavily 를 **추가 1~2회** 호출한다 (총 호출이 ≤8 가 되도록 스킬이 제어하지만, 넘어서더라도 스크립트가 exit 4 로 차단한다). 결과를 6단계와 동일 형식으로 요약 출력한다.

### 9단계: Q&A 생성 (검색 결과에 맞춰 동적 생성, 검색 스킵 시 고정 fallback)

#### 9-A. 검색 경로 (쿼리 1개 이상 실행된 경우) — **카테고리·질문 동적 생성**

AI 가 **주제 + Tavily 결과 + 프로파일** 을 종합해 **4~6개 카테고리**와 각 카테고리당 **2~3개 질문** (총 8~12문) 을 즉석 생성한다. 목적은 "검색이 이미 답한 질문을 또 묻지 않고, 검색이 드러낸 구체 각도를 찌르는 것".

**카테고리 구성 규칙 (필수 제약)**:
- **CLAUDE.md 섹션으로 매핑 가능해야 한다.** 각 카테고리는 내부적으로 `→ 기술설명` / `→ 선택섹션` / `→ 경험` / `→ 마치며` 중 하나에 1:1 대응된다. 대응은 스킬이 내부 메모로만 갖고, 사용자에게는 카테고리 이름만 보인다.
- `→ 기술설명` 매핑 카테고리 **1~2개**, `→ 경험` 매핑 카테고리 **1~2개**, `→ 마치며` 매핑 카테고리 **1개** 는 필수. `→ 선택섹션` 매핑 카테고리는 **0 또는 1개**.
- 카테고리 이름은 주제/검색 결과에 맞게 자유롭게 짓는다. 예: 트렌드성 주제면 `[지금 뭐가 핫한가]` · `[밑에서 뭐가 돌아가나]` · `[현장 반응]` · `[내 시각]`. 튜토리얼성 주제면 `[왜 쓰나]` · `[어떻게 동작하나]` · `[흔한 함정]` · `[경험]` · `[한 문장 요약]`.

**질문 구성 규칙**:
- **검색이 강하게 답한 질문**: AI 가 검색 근거로 **초안 답**을 먼저 작성 → "이 초안을 수정/보강해주세요" 로 요청. 사용자 부담 ↓.
- **사용자 고유 경험이 필요한 질문**: 빈 칸으로 둠 (예: "이거 쓰면서 네가 겪은 가장 큰 삽질은?").
- **검색 결과가 드러낸 구체 각도**를 질문 1~2개에 반드시 반영. 예: 검색이 "uv 1.0 발표가 pip-tools 생태계 흔듦"을 드러냈다면, 일반적 "대안 비교?" 대신 "pip-tools 쓰다가 uv 로 옮긴다면 가장 걸리는 지점?" 식.

**출력 형식** (예시, 주제: "uv 패키지 매니저"):

```
주제: uv 패키지 매니저 (프로파일: hmkim)
검색 요약 기반으로 카테고리와 질문을 짰습니다. 빈 칸만 답해주셔도 되고, 초안은 수정 환영.

[지금 왜 uv인가] (→ 기술설명 도입)
Q1. 초안: "2024년 말 astral-sh 가 uv 를 발표하면서 pip 대비 10~100배 빠른 Rust 기반 매니저가 등장. 최근 uv 1.0 발표로 프로덕션 사용 가능." — 이 맥락에서 본인이 왜 이걸 다루고 싶었는지 한 줄 추가?
Q2. (빈 칸) 네가 처음 uv 를 접한 순간·계기는?

[어떻게 동작하나] (→ 기술설명 본문)
Q3. 초안: "Rust 기반 resolver + 단일 바이너리 + PEP 723 인라인 스크립트 지원" — 이 3가지 중 사용자가 제일 체감하는 포인트 하나 꼽으면?
Q4. (빈 칸) 코드/명령 예시 하나만 보여줄 수 있나? (uv init / uv run / pyproject 같은 것)

[pip·poetry 와 뭐가 다른가] (→ 선택섹션: 대안 비교)
Q5. 초안: "pip(표준·느림) / poetry(pyproject·의존성 락) / uv(전부 합친 Rust 구현)" — 이 비교가 맞나? 네 경험상 다른 포인트?

[내가 겪은 uv] (→ 경험)
Q6. (빈 칸) uv 로 바꾼 뒤 생긴 변화 중 가장 큰 것은?
Q7. (빈 칸) 막혔던 지점 / 실수했던 경험?

[다음 문장으로 남기고 싶다면] (→ 마치며)
Q8. (빈 칸) 이 글을 덮은 독자가 한 줄로 기억했으면 하는 메시지는?
```

답이 빈 항목은 "[작성 필요]" 로 표시하고 초안 단계로 넘어간다.

#### 9-B. 검색 스킵 경로 (쿼리 0개 선택되어 Tavily 호출 없음) — **고정 9문 Q&A verbatim**

검색 결과가 없으면 동적 카테고리 생성 근거가 없으므로 고정 9문 Q&A 로 fallback. 아래 텍스트와 **정확히 동일**하게 출력한다:

```
주제: {주제}

글을 풍부하게 쓰려면 몇 가지 물어봐야 한다.

[맥락]
Q1. 이 주제를 다루게 된 계기가 뭔가? (어떤 문제를 만났나, 어떤 필요가 있었나)
Q2. 이 주제가 지금 왜 중요한가? (배경, 트렌드, 실무에서 왜 필요해졌나)

[기술 핵심]
Q3. 기술적으로 핵심 포인트 1~3가지를 꼽는다면?
Q4. 이 기술과 비교할 만한 대안/경쟁 기술이 있나? 있다면 어떻게 다른가?
Q5. 대표적인 코드/명령어/설정 예시를 하나 보여줄 수 있나? (없으면 스킵)

[경험]
Q6. 이 주제를 다루면서 막혔거나 삽질했던 경험이 있나?
Q7. 그 문제를 어떻게 해결했나?
Q8. 입문자가 흔히 빠지는 함정이나 실전 팁 1~3가지는?

[메시지]
Q9. 이 글을 읽는 입문자에게 가장 전달하고 싶은 한 문장은?
```

답이 빈 항목은 "[작성 필요]" 로 표시하고 진행한다.

### 10단계: 선택 섹션 결정 (경로별)

**검색 경로 (9-A)**: 9-A 에서 이미 `→ 선택섹션` 매핑 카테고리를 0 또는 1개로 정해두었으므로 **추가 판단 불필요**. 그대로 초안 섹션에 반영.

**검색 스킵 경로 (9-B)**: 기본 1-a 로직 그대로 적용.
- Q2 답변이 풍부하면 → **"왜 지금 이 주제인가"** 섹션 추가 고려
- Q4 답변에 비교 대상이 2개 이상이면 → **"대안 비교"** 섹션 추가 고려 (표 필수)
- Q8 답변이 3개 가까이 구체적이면 → **"실전 팁 / 함정 피하기"** 섹션 추가 고려 (체크리스트)

셋 중 **최대 1개만** 선택. 모두 약하면 선택 섹션 없이 기본 2섹션으로 간다.

### 11단계: 초안 작성

답변 + 검색 결과를 토대로 CLAUDE.md 규칙을 따르는 초안을 만든다. **CLAUDE.md 가 단일 정관** — 문체·분량·도입부·제목·가독성·금지 표현·영어 표기·마무리 화이트리스트 모두 그쪽이 진리.

CLAUDE.md 외에 이 단계에서만 추가로 강제하는 항목:

- **체크리스트(`- [ ]`) 금지**, **비교 표(`| --- |`) 금지** (CLAUDE.md 분량 절에 가독성 보조 도구로 쓰지 말 것)
- 코드 블록 1개 이상 (코드가 의미 있는 주제에 한함)
- **인라인 출처 표기**는 다음 3가지 케이스에만 — `(출처: URL)` 형태로 본문 안에 박는다:
  - (a) 공식 발표·공식 문서 인용
  - (b) 원문 직접 인용
  - (c) 최신성 주장 (출시일·버전 등)
- 평범한 설명 문장에는 출처 X. 글 끝 References 섹션도 X.
- **검색 결과는 참고 자료**다. 커뮤니티 반응·블로그 주장은 `~라는 반응이 있다 / ~로 알려졌다` 식 완곡 표현. 출처 없는 수치는 표기하거나 빼라. 거짓말보다 빈 문장이 낫다.
- 도입부 두 단락(SEO 정의 + 1인칭 후크) 형식·길이·금지 사항은 CLAUDE.md `## 도입부` 절 그대로 따른다. **첫 단락에 1인칭 어투(`나는 / 내가 / 처음엔`) 절대 금지**.

### 12단계: 파일 저장 + 본문 이미지 마커 + 꼬리 블록

`posts/YYYY-MM-DD-<슬러그>.md` 로 저장 (오늘 날짜).

**파일 전체 구조** (위→아래 순):

```
# H1 제목

![이미지: 썸네일](...)   ← thumb 마커 (H1 바로 아래)

도입 단락 1 — SEO 정의 한 줄 (100~160자, 1인칭 금지, 평서체)

도입 단락 2 — 1인칭 후크 (2~4문장)

<!-- 본문 광고 -->        ← 사용자가 티스토리 업로드 후 광고 코드 수동 삽입

## 본문 H2 (1)
![이미지: section-1](...)
본문…
## 본문 H2 (2)
![이미지: section-2](...)
본문…
... (본문 H2 추가 시 section-N 마커 동일하게)
## (마무리 H2 — 화이트리스트)
마무리 (이미지 마커 없음)
## 태그
콤마/공백 구분 태그 목록
## 이미지 메타
YAML 블록 (/blog-cards 입력)
```

도입부 두 단락은 `## 도입부` 같은 H2 제목 없이 썸네일 바로 아래에 그대로 둔다. 단락 사이 빈 줄 1개로 분리. 두 단락 다음 줄에 `<!-- 본문 광고 -->` HTML 주석 한 줄을 두고 첫 H2 로 넘어간다.

**본문 내 이미지 마커** — **"마무리 H2 제외한 본문 H2 수 + 1(thumb)" 만큼**, 최대 **5장**:

- `thumb` → **H1 바로 아래**, 도입 문단 **앞** (항상 1장). 검색엔진이 이미지를 건너뛰고 다음 텍스트(SEO 정의 한 줄) 를 description 으로 잡는다.
- `section-N` (N=1,2,...) → 마무리 제외 본문 H2 각각 바로 **아래**
- 본문 H2가 5개를 넘으면 처음 4개에만 카드 (`section-1`~`section-4`)
- 마무리 섹션(`## 마치며` 등) 에는 카드 없음

예 — 본문 H2가 3개(기술/선택/경험)면 카드 4장 (thumb + section-1,2,3):

```markdown
![이미지: 썸네일](images/YYYY-MM-DD-슬러그/thumb.webp)
![이미지: section-1](images/YYYY-MM-DD-슬러그/section-1.webp)
![이미지: section-2](images/YYYY-MM-DD-슬러그/section-2.webp)
![이미지: section-3](images/YYYY-MM-DD-슬러그/section-3.webp)
```

경로는 마크다운 파일 기준 상대경로. 티스토리 업로드는 수동이라 실제 배포 시엔 드래그로 덮어쓰지만, 로컬 미리보기·작성자 시각 확인 용도로 자리를 잡아둔다.

**`## 태그` 블록** (마치며 바로 다음):

티스토리 태그로 쓸 키워드 6~10개를 한 줄에 콤마로 구분해 적는다. 주제·핵심 용어·플랫폼 이름 중심. 예:

```markdown
## 태그

Claude, Claude Code, 요금제, Pro, Max 5x, Max 20x, AI 구독, 개발 생산성
```

**`## 이미지 메타` 블록** (태그 다음, 파일 맨 끝, `/blog-cards` 입력): 아래 13단계에서 정의한 YAML 블록을 둔다.

### 13단계: 이미지 메타 블록 생성 (카드 템플릿용)

`## 태그` 블록 바로 아래에 `## 이미지 메타` YAML 블록을 둔다. `/blog-cards` 가 이 블록을 읽어 카드형 WebP 를 만든다.

**블록 구성**:
- 최상단 `palette:` (포스트 단일, hex 3색)
- `thumb` (1개) + `section-1` ~ `section-N` (마무리 제외 본문 H2 수와 동일, N≤4)
- 카드 총 수 1+N, 캡 5장

**팔레트 — 글 톤 보고 hex 3색 직접 정한다**:
- 프리셋·고정값 금지. 매 글마다 base/accent/highlight 새로 결정.
- 어떤 hex 가 어울리는지 판단 기준은 `/blog-cards` SKILL.md 팔레트 절 참조.
- **이전 글의 hex 를 그대로 가져오지 않는다.** 직전 글과 색이 겹치면 변형.

**템플릿·필드 명세는 `/blog-cards` SKILL.md 단일 정관**:
- 썸네일 4종(`thumbnail`/`-bold`/`-stat`/`-quote`)
- 섹션 8종(`section-hero`/`-simple`/`-numbered`/`-timeline`/`-split`/`-stat-grid`/`-compare`/`-callout`)
- 각 템플릿이 어떤 H2/H3 패턴에 맞는지, 어떤 필드를 받는지 모두 그쪽에 있다.

**고르는 순서 (요약)**:
1. H2 제목·톤 → `thumb` 4종 중 하나
2. 각 본문 H2 의 H3 개수·키워드·수치 분포 → `section-N` 8종 중 하나
3. 애매하면 `thumbnail` + `section-hero`(나열) / `section-simple`(결론) 기본 조합

**출력 yaml 골격** (구체 hex·문구는 매번 글에 맞춰 새로 — 아래 placeholder 그대로 복사 금지):

```yaml
palette:
  base: "#<글 톤에 맞는 어두운 hex>"
  accent: "#<감정·강조 hex>"
  highlight: "#<base 위에서 떠 보이는 밝은 hex>"

thumb:
  template: <thumbnail|thumbnail-bold|thumbnail-stat|thumbnail-quote>
  # 템플릿별 필드는 /blog-cards SKILL.md 참조

section-1:
  template: <section-* 8종 중 하나>
  # 필드 동일
```

**문구 길이 가이드** (LLM 이 자주 빠뜨리는 부분만):
- 카드 제목·배지 8~14자. 더 길면 카드에서 잘림.
- badges 4~6개 권장.
- section-compare 는 `emph: true` 를 권장·결론 쪽에.

## Tavily 호출 예산 (하드 상한)

1회 실행 총 Tavily 호출 ≤8. 이는 LLM 권고가 아니라 `scripts/tavily-search.py` 가 `BLOG_RESEARCH_SESSION_ID` 기반 세션 카운터(`.omc/state/tavily-budget.json`)로 코드 레벨에서 강제한다. 9번째 호출은 `[error] budget exceeded (8/8)` 와 함께 exit 4 로 차단된다.

## 인용 표기 규칙 요약

| 케이스 | 예시 | 인라인 출처 |
|--------|------|------------|
| 구체적 수치 | "설치 속도 10배" | O |
| 원문 인용 | "공식 문서는 X라 한다" | O |
| 최신 발표/출시 | "2026년 X 출시" | O |
| 평범한 설명 | "A는 B와 유사하다" | X |

## 에러 대응

- `TAVILY_API_KEY` 없음 → 3단계에서 바로 중단, 파일 생성 없음.
- `BLOG_RESEARCH_SESSION_ID` 미주입 (버그) → 스크립트가 exit 5. 스킬이 세션 ID 설정 단계를 누락한 것이므로 0단계를 다시 실행.
- 예산 초과 → 스크립트 exit 4. 사용자에게 "한 번의 run 당 ≤8 호출 제한을 넘었다" 안내 후 심화 검색 건너뛰고 Q&A 로 진행.
- Tavily 429 → 스크립트가 15초 대기 후 2회 재시도. 최종 실패 시 스킬은 해당 쿼리를 스킵하고 사용자에게 알림.
- 사용자가 쿼리 0개 선택 → 검색 없이 Q&A 만으로 초안. 인라인 출처 0개여도 정상 저장.
