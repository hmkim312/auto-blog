블로그 한 편을 끝까지 만드는 메인 파이프라인. 영문 자료를 LLM 한 번에 한국어 글로 옮기면 어순·동사 결합이 영어식으로 새어 나오므로 **번역과 작문을 분리한다**. 메인 Claude (Opus 4.7) 가 영문 Tavily 결과를 한국어 사실 카드로 옮긴 뒤 같은 Opus 가 카드만 보고 본문을 쓰고, Opus sub-agent 가 다른 컨텍스트에서 검수한다.

CLAUDE.md 가 글쓰기 룰의 단일 정관. 이 파일은 절차만 적는다.

## 사용법

```
/blog                          # 주제 탐색부터
/blog uv 패키지 매니저          # 주제 직접 지정 (0단계 스킵)
/blog --profile hmkim          # 프로파일 적용 + 탐색
/blog uv --profile hmkim       # 둘 다
```

## 0단계: 주제 결정

주제 인자가 있으면 스킵. 없으면 `blog-discover` 흐름 그대로 — `scripts/discover-topics.py` → 후보 10 개 → 사용자 선택.

## 0.5단계: 핵심 키워드 + 슬러그 (SEO)

CLAUDE.md `## 11. SEO` 룰. 한국 검색 수요 기준 키워드 후보 3~5 개 추천 → 사용자 선택 → 슬러그 → 파일명 `posts/YYYY-MM-DD-<슬러그>.md` 확정.

이 키워드는 이후 H1·도입 첫 단락·썸네일 alt 까지 같은 형태로 일관.

## 1단계: 검색 + Q&A

### 세션·프로파일·키 확인

- 세션 ID: `BLOG_RESEARCH_SESSION_ID=blog-$(date +%Y%m%d-%H%M%S)-$$`. 이후 Tavily 호출 전부에 환경변수로 전달. 세션당 ≤8 회 하드 상한 (스크립트가 코드 레벨로 강제).
- 프로파일: `--profile <name>` 있으면 `profiles/<name>.yml` Read. 없으면 기본값 (`reader_level: AI 에 관심 있는 모든 일반인`).
- `.env` 의 `TAVILY_API_KEY` 없으면 이 시점에 중단.

### AI 쿼리 제안 → 사용자 승인 → Tavily 검색

주제 + 프로파일로 5~7 개 쿼리 후보를 번호 리스트로 제시 (서로 다른 각도: 정의·트렌드·비교·실사용·성능·함정). 사용자가 번호 선택·편집·추가 → 최종 ≤6 개 확정. 0 개 선택 시 검색 스킵.

확정된 쿼리를 순차로:

```bash
BLOG_RESEARCH_SESSION_ID=<ID> uv run scripts/tavily-search.py \
  --query "<쿼리>" --max-results 10 --include-answer \
  [--language ko --region KR]
```

`--language`/`--region` 은 프로파일 `search_bias` 있을 때만.

### 심화 검색 (선택)

"더 파고 싶은 키워드 있나?" → 있으면 추가 1~2 회 (총 ≤8).

### Q&A — 검색 결과 기반 동적 생성

검색이 1 회 이상 돌면 AI 가 주제 + Tavily 결과 + 프로파일로 **4~6 개 카테고리, 카테고리당 2~3 개 질문** (총 8~12 문) 즉석 생성. 검색이 강하게 답한 질문은 AI 가 초안 답을 먼저 쓰고 "수정/보강 환영", 고유 경험은 빈 칸.

카테고리 내부 매핑 (사용자에게 보이진 않음): `기술설명` 1~2, `경험` 1~2, `마치며` 1 필수, `선택섹션` 0~1.

**검색 스킵 시 fallback — 고정 9 문 Q&A** (verbatim):

```
[맥락]
Q1. 이 주제를 다루게 된 계기가 뭔가?
Q2. 이 주제가 지금 왜 중요한가?

[기술 핵심]
Q3. 기술적으로 핵심 포인트 1~3 가지를 꼽는다면?
Q4. 비교할 만한 대안/경쟁 기술이 있나? 어떻게 다른가?
Q5. 대표 코드/명령어/설정 예시 하나? (없으면 스킵)

[경험]
Q6. 막혔거나 삽질했던 경험?
Q7. 어떻게 해결했나?
Q8. 입문자가 흔히 빠지는 함정·실전 팁 1~3 가지?

[메시지]
Q9. 입문자에게 가장 전달하고 싶은 한 문장?
```

답 빈 항목은 `[작성 필요]` 로 표시하고 진행.

### 선택 섹션 결정

- 동적 Q&A 경로: 카테고리 매핑에 이미 `→ 선택섹션` 1 개 또는 0 개로 정해져 있음.
- 고정 9 문 경로: Q2 답 풍부 → "왜 지금 이 주제인가" / Q4 비교 대상 2+ → "대안 비교" / Q8 구체 3 개 → "실전 팁·함정". 최대 1 개.

## 1.5단계: 영문 → 한국어 사실 카드 변환

메인 Claude 가 직접. 사실·수치·인용만 옮긴다 (의견·해석 X). 한 사실 = 한 카드 = 한국어 한 문장 + 출처 URL.

CLAUDE.md `## 2. 번역과 작문 분리` 룰. 영어 수동태 → 능동형, 영어식 명사구 → 동사형.

```markdown
## 한국어 사실 카드

### 카드 1: 정의·등장 배경
- 사실: ...
- 시점: ...
- 출처: ...

### 카드 N: ...
```

Tavily 결과 중 본문에 들어갈 사실만 추려 5~10 개. 사용자 Q&A 답변도 같은 형식으로 함께 (1인칭 보존).

이 카드 묶음이 2-A 입력. 카드에 없는 사실·수치·인용·URL 은 본문에 만들지 않는다. 개념 설명·비유·맥락은 Opus 본인 지식으로 보강 OK.

## 2단계: 작성 + 검수 (둘 다 Opus, 다른 컨텍스트)

### 2-A. Opus 작성 (메인 Claude 직접)

메인 Claude 가 한국어 사실 카드 + 사용자 Q&A 만 보고 본문 작성. sub-agent 위임 X. **영문 Tavily 원문은 컨텍스트에서 의식적으로 제외** — 영어 어순에 다시 노출되면 1.5단계가 무의미.

#### 입력 소스 우선순위

1. 사실·수치·인용·URL → 카드 + Q&A 가 정답. 없는 통계·인용·URL 만들지 않는다.
2. 개념·배경·비유·예시·코드·맥락 → Opus 본인 지식 능동 활용.
3. 사용자 1인칭 경험 → Q&A 답변 그대로. 없는 경험 윤색 X.

본인 지식 보강 가드:
- 단정형 수치는 출처가 카드에 있어야 함. 없으면 정성 표현 (`알려져 있다`).
- 카드와 본인 지식 충돌 시 카드 우선 (최신).
- 의심 주장은 `~ 라고 알려져 있다`, `~ 라는 분석이 있다` 완곡.

#### 작성 룰 (CLAUDE.md 발췌)

CLAUDE.md `## 1` ~ `## 18` 전체 적용 (글쓰기 핵심·번역투·골든 톤·도입부·제목·구조·문체·영어 용어·SEO·분량·가독성·인용·카드·X 요약·파일명·반구어체). 어색 어휘는 `awkward-words.md`.

추가로 이 단계에서만:
- H1 에 0.5단계 키워드 자연 포함 + 도입 첫 단락에도 한 번. H2·H3 까지는 강박 X.
- 체크리스트 (`- [ ]`) X, 비교 표 (`| --- |`) X.
- 코드 블록 1 개 이상 (코드가 의미 있는 주제 한정).
- 인라인 출처 `(출처: URL)` — 공식 발표·원문 인용·최신성 주장만.
- 본문 마크다운 `![alt](path)` 의 alt 는 3단계에서 정할 카드 alt 와 동일.

#### 파일 전체 구조

```
# H1

![썸네일 alt](images/<슬러그>/thumb.webp)

도입 단락 1 (정의 또는 상황 제시, 100~180 자)

도입 단락 2 (1인칭 후크)

## 본문 H2 (1)
![section-1 alt](images/<슬러그>/section-1.webp)
H3...

## 본문 H2 (2)
![section-2 alt](images/<슬러그>/section-2.webp)
H3...

## (마무리 H2 — 이미지 마커 없음)
마무리

## 태그
키워드 콤마 구분

## 인터럴 링크
나가는 / 들어오는

## 이미지 메타
YAML 블록 (3단계에서 추가)
```

본문 이미지 마커 수: 썸네일 1 + (마무리 제외 본문 H2 수). 최대 5 장.

#### 자가 점검 (저장 전)

1. 핵심 원칙 4 가지 (결론 먼저·한 문장 한 내용·능동형·숫자 받침) 다 적용?
2. `awkward-words.md` 의 어색 어휘가 본문에 어색하게 등장하지 않는가?
3. 영어 수동태 직역 없는가?
4. H1·H2·H3 명사형 강제 비틀기 없는가?
5. 카드 사실 그대로 반영? (윤색 X)
6. Q&A 에 없는 도구·가드·단계 끼어 들어가지 않았나?
7. 인라인 출처·ILJ 마커·이미지 마커 다 들어갔나?
8. H1·도입에 후크 (구체 수치·갈등·결과) 살아 있나?

저장 후 본문 텍스트는 사용자에게 출력 X. "초안 작성 완료 (Opus), Opus 검수로 진행" 만 알린다.

### 2-B. Opus 검수

`Agent` 툴, `subagent_type: claude`, `model: opus`. 다른 컨텍스트라 자기 합리화 분리.

메인 Claude 가 프롬프트에 CLAUDE.md 발췌 + blog-review SKILL.md 결정론 룰 (10-e) + 자연 한국어 메모리를 사전 주입해서 sub-agent 가 cat 으로 다시 안 읽게.

```
한국어 블로그 검수자다. 다음 글을 평가하고 JSON 리포트만 출력.

파일: posts/YYYY-MM-DD-<슬러그>.md

[평가 룰 — CLAUDE.md 발췌 + blog-review 결정론 + 사용자 메모리 전부 인라인]

SKIP 항목 (3단계 처리):
- 썸네일 마커 부재 / section-N 카드 마커 부재
- 이미지 alt 자리채움 / 중복
- `## 이미지 메타` YAML 부재

평가 항목:
1. 결정론 위반 (문체·이모지·도입부·em-dash)
2. `awkward-words.md` 어색 어휘 검출 — 각 단어 등장 시 줄·원문 인용 + 맥락이 자연스러운지 짚는다 (대체어 추천 X)
3. 자연 한국어 (영어 어순·명사화·격언·평론·메타 멘트)
4. 후크 (H1·도입에 구체 수치·갈등)
5. 사실 출처 의심
6. Q&A 에 없는 도구·가드 윤색
7. 반구어체 톤 (CLAUDE.md `## 18`)
8. X 요약 평가 (있으면)

무조건 N 건 이상 짚을 것 (자기 confirmation bias 회피). 'OK' 한 줄 X.

출력:
{
  "score": <0-100>,
  "verdict": "PASS" | "FAIL",
  "결정론_위반": [...],
  "어색_어휘": [{"line":N,"원문":"...","어휘":"...","판정":"..."}, ...],
  "자연_한국어": [...],
  "후크_평가": "...",
  "사실_출처_의심": [...],
  "윤색_의심": [...],
  "반구어체_톤": [...],
  "x_요약_평가": "..."
}

PASS 컷오프: score ≥ 90 + 결정론 위반 0 + 어색 어휘 '어색' 판정 0.
```

### 2-C. Opus 재작성 1 회 (FAIL 시만)

`verdict == FAIL` 이면 메인 Claude (Opus) 가 JSON 위반 항목 전체 반영해 본문 1 회 재작성. 재검수 자동 X. `/goal` 무한 루프 시에만 2-B → 2-C 반복.

### 2-C.5 어색 어휘 카운트 누적

2-B 리포트의 `어색_어휘` 배열에서 `판정: 어색` 인 항목, 그리고 이후 사용자가 직접 지적한 어색 표현마다 `awkward-words.md` 의 해당 단어 행 카운트 `(N)` 를 +1. 리스트에 없는 신규 어휘는 `- <어휘> (1)` 로 새 줄 추가. 같은 글에서 같은 어휘 여러 줄 검출은 줄 수만큼 누적.

### 2-D. 루프 출력

```
✅ 본문 단계 완료
- Opus 작성 완료 (N 자)
- Opus 검수: <PASS | FAIL, score N/100>
- Opus 재작성: <적용 | 스킵>
주요 지적: <어색 N 건, 자연 한국어 N 건, 사실 N 건>
다음 단계: 카드 메타 생성
```

## 3단계: 카드 메타 자동 생성

사용자 확인 X.

- 팔레트 hex 3 색 (글 톤에 맞게 새로. 직전 글과 겹치면 변형. 프리셋 X).
- 카드 수: thumb 1 + section-N (마무리 제외 본문 H2 수, 최대 4).
- 카드 타이틀은 H2 메시지와 정렬 (CLAUDE.md `## 15. 카드 콘텐츠`).
- alt: 30~80 자 한국어. thumb alt 에만 핵심 키워드. section-N alt 는 해당 H2 메시지 중심. 카드 간 중복 X.
- 본문 마크다운 `![alt](path)` 의 alt 도 Edit 로 동기화.

`## 태그` 다음에 `## 이미지 메타` YAML 블록 추가. 양식·필드 명세는 `blog-cards/SKILL.md`.

## 4단계: 카드 렌더

```
uv run scripts/render-cards.py posts/<파일>
```

## 5단계: X 아티클 요약

CLAUDE.md `## 16. X 아티클 요약` 룰. 메인 Claude 가 후킹 요약 1 개 생성, 본문 파일 `## 이미지 메타` 다음에 추가:

```markdown
## X 아티클 요약

<후킹 요약 1~2 줄>
```

후보 3 안 X (자동 흐름이라 1 개 확정).

## 시간 예상

- 1단계 (검색·Q&A): 사용자 인터랙션 비례
- 1.5단계 (사실 카드): 1~2 분
- 2단계 (Opus 작성 + Opus 검수 + 재작성 1 회): 3~6 분
- 3~5단계: 1~2 분

총 5~10 분 (`/goal` 무한 루프 시 가변).

## 에러 대응

- `TAVILY_API_KEY` 없음 → 1단계 시작 시 중단.
- Tavily 예산 초과 → 스크립트 exit 4 + 사용자 알림 → 심화 검색 건너뛰고 Q&A.
- 사용자가 0단계 중단 → 1.5단계 진행 X.
- 2-B Opus sub-agent 실패 → 1 회 재시도, 그래도 실패 시 사용자 수동 진행 (메인 Opus 같은 컨텍스트 자가 검수 금지).
- 카드 렌더 실패 → 해당 카드만 재시도: `/blog-cards <파일> --only <key>`.

## 규칙

- 본문 작성·검수 둘 다 Opus. 같은 컨텍스트 자가 검수만 금지 (자기 합리화).
- 영문 원문은 사실 카드 거친 뒤에만 작성 단계로.
- `blog-review --fix` 자동 교정은 이 파이프라인에서 사용 X (`--judge` JSON 만).
